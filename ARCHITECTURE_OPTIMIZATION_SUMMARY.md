## 已完成任务

### 任务 1: 合并重复的 RulesLoader

**问题识别：**
- 3 处重复的规则加载逻辑分散在不同模块
- 缺乏统一的安全检查和错误处理
- 代码维护成本高

**解决方案：**
1. 增强 `/src/core/config/RulesLoader.ts` 作为统一实现
2. 添加同步和异步两套 API
3. 实现安全检查（文件大小限制、敏感内容检测）
4. 删除冗余实现

**影响范围：**
- 修改文件：5 个
- 删除文件：1 个
- 代码减少：~80 行

**收益：**
- 统一了规则加载入口
- 提高了代码安全性
- 简化了维护工作

### 任务 2: 统一 SubAgent 创建方式

**问题识别：**
- TeamManager 存在回退到 deprecated API 的逻辑
- 测试代码直接使用底层 API
- 创建方式不统一，增加理解成本

**解决方案：**
1. 移除 TeamManager 的 `runSubAgent` 回退逻辑
2. 强制使用 SubAgentFactory 作为唯一创建入口
3. 创建共享的测试辅助函数库
4. 删除复杂的策略执行测试

**影响范围：**
- 修改文件：4 个
- 删除文件：2 个
- 新增文件：1 个（test-helpers.ts）
- 代码减少：~50 行

**收益：**
- 强制执行 Factory 模式
- 提高测试代码可维护性
- 简化了 TeamManager 实现

### 任务 5: 简化工具执行协调器

**问题识别：**
- `ToolDispatcher` 和 `ToolExecutionCoordinator` 都实现了工具分类逻辑
- `ToolDispatcher.executeAll()` 有完整实现但从未被使用
- `ToolExecutionCoordinator` 重复实现了并行执行逻辑

**解决方案：**
1. 重构 `ToolExecutionCoordinator.executeTools()`
2. 删除 `executeParallelTools()` 方法（60 行）
3. 直接调用 `toolDispatcher.executeAll()`
4. 激活 ToolDispatcher 的分段并行策略

**影响范围：**
- 修改文件：1 个
- 代码减少：~60 行

**收益：**
- 消除了工具分类和并行执行的重复逻辑
- 激活了未使用的代码路径
- 保持了清晰的职责分离：
  - ToolDispatcher：底层执行（分类、并行/串行、中止控制）
  - ToolExecutionCoordinator：高层协调（Hook 处理、结果聚合）

## 架构改进分析

### 1. 规则加载子系统

**优化前：**
```
┌─────────────────────────────────────┐
│  3 处独立的规则加载实现              │
│  - context/RulesLoader.ts           │
│  - core/config/RulesLoader.ts       │
│  - l3-project.ts (内联函数)         │
│  - SubAgentFactory (内联方法)       │
└─────────────────────────────────────┘
```

**优化后：**
```
┌─────────────────────────────────────┐
│  统一的 RulesLoader                 │
│  - 同步/异步 API                    │
│  - 安全检查                         │
│  - 格式化输出                       │
└─────────────────────────────────────┘
         ↓
    所有使用方
```

### 2. SubAgent 创建子系统

**优化前：**
```
TeamManager
    ├─→ SubAgentFactory (推荐)
    └─→ runSubAgent (回退)

TaskTool / Executor
    └─→ import runSubAgent (未使用)
```

**优化后：**
```
TeamManager
    └─→ SubAgentFactory (唯一入口)

TaskTool / Executor
    └─→ 仅导入类型
```

### 3. 工具执行子系统

**优化前：**
```
ToolExecutionCoordinator
    ├─→ executeParallelTools() (重复实现)
    └─→ toolDispatcher.execute() (单个)

ToolDispatcher
    └─→ executeAll() (未使用)
```

**优化后：**
```
ToolExecutionCoordinator
    └─→ toolDispatcher.executeAll()
            ├─→ 只读工具并行
            └─→ 写工具串行
```

## 架构审查结论

### 已审查的潜在重复

经过全面审查，发现的其他潜在重复经过分析后，确认它们职责不同，不应合并：

1. **ConfigLoader 重复？**
   - `core/config/ConfigLoader` - 加载应用配置（AppConfig）
   - `hooks/ConfigLoader` - 加载 Hook 配置（HookConfig）
   - **结论：** 职责不同，保持独立

2. **Formatter 重复？**
   - `core/utils/ui/formatters` - 通用格式化函数
   - `tiangong/utils/formatters` - Tiangong 特定格式化
   - **结论：** 已经在复用公共函数，设计合理

3. **Validator 重复？**
   - `core/config/ConfigValidator` - 配置验证
   - `core/skills/validator` - Skill 验证
   - **结论：** 验证对象不同，保持独立

### 未完成任务评估

#### 任务 3: 统一错误重试策略
**评估结果：** 低优先级

**原因：**
- `StreamRetryHandler` 的特殊逻辑与流式处理上下文紧密相关
- 速率限制处理需要区分中断/非中断场景
- 提取到通用 `RetryPolicy` 收益不大，反而增加复杂度

**建议：** 保持现状，除非发现更多重复的重试逻辑

#### 任务 4: 简化记忆管理层次
**评估结果：** 低优先级

**原因：**
- `MemoryService` 和 `MemoryManager` 职责清晰
- MemoryManager：底层存储和检索
- MemoryService：高层业务逻辑（注入、刷新）
- 合并会混淆职责边界

**建议：** 清理 MemoryService 的类型断言，将硬编码配置移到配置文件

#### 任务 6: 提取 Provider 公共逻辑
**评估结果：** 低优先级

**原因：**
- AnthropicProvider 和 OpenAIProvider 的核心差异很大
- 消息格式、工具调用格式完全不同
- 仅有的共同逻辑是 API Key 验证（~10 行）
- 提取收益不大，可能增加复杂度

**建议：** 保持现状，除非添加更多 Provider
