# 工具 Description 改进

## 改进目标

工具的 `description` 应该清晰地引导 agent 如何正确使用，而不是将使用规则硬编码在主 agent 的 prompt 中。

## 设计原则

1. **工具自描述**：每个工具的使用方法由工具自己的 `description` 定义
2. **结构化说明**：使用清晰的章节结构（用途、使用流程、示例、注意事项）
3. **完整示例**：提供正确和错误的使用示例
4. **分步指导**：明确说明使用步骤和决策点
5. **通用行为规范**：主 agent 的 prompt 只包含通用的行为规范，不涉及具体工具

## 改进内容

### 1. TaskTool - task 工具

**改进前**：
- 混合了英文和中文
- 信息分散，不够结构化
- 缺少清晰的使用流程

**改进后**：
```
# task - 委派任务给子 agent

## 用途
将复杂任务委派给专门的子 agent 独立执行。

## 使用流程（必须遵循）

### 步骤 1：查找合适的 agent
首先调用 match_agent 查找最合适的预置 agent

### 步骤 2：根据匹配结果决定如何调用
- 情况 A：找到合适的预置 agent（匹配分数 >= 0.5）
- 情况 B：没有合适的预置 agent（匹配分数 < 0.5）

## 临时 agent 的要求
- system_prompt 编写要点
- tools 选择指南

## 完整示例
- 示例 1：使用预置 agent
- 示例 2：创建临时 agent

## 常见错误
- 错误 1：直接使用不存在的 agent
- 错误 2：创建临时 agent 但缺少参数
- 错误 3：跳过 match_agent 直接创建临时 agent
```

**关键改进**：
- 使用 Markdown 标题结构化内容
- 明确的两步流程：先 match_agent，再 task
- 清晰的决策点：根据匹配分数决定使用预置还是临时 agent
- 完整的正确和错误示例
- 工具选择指南（根据任务类型）

### 2. MatchAgentTool - match_agent 工具

**改进前**：
- 只说明了工具的功能
- 没有说明如何使用返回结果

**改进后**：
```
# match_agent - 查找最合适的 agent

## 用途
根据任务描述，自动推荐最合适的预置 agent。

## 何时使用
在调用 task 或 agent_team 工具之前，必须先调用此工具。

## 使用流程

### 步骤 1：调用 match_agent
### 步骤 2：查看推荐结果
### 步骤 3：根据分数决定
- 如果最高分数 >= 0.5：使用推荐的预置 agent
- 如果最高分数 < 0.5：创建临时 agent

## 完整示例
- 示例 1：找到合适的预置 agent
- 示例 2：没有合适的预置 agent

## 参数说明
## 注意事项
```

**关键改进**：
- 明确说明何时使用（在 task 之前）
- 清晰的三步流程
- 明确的决策点：0.5 分数线
- 完整的示例展示如何使用返回结果

### 3. MainAgent System Prompt

**改进前**：
- 只有 5 条原则
- 没有错误处理指导

**改进后**：
在 "## 原则" 部分添加了第 6 条：
```
6. **工具调用失败时的处理**：
   - 仔细阅读错误信息，理解失败原因
   - 根据错误提示调整参数后重试
   - 最多重试 2 次
   - 如果重试仍然失败，向用户说明失败原因和尝试过的解决方案
   - 不要在第一次失败后就直接终止任务
```

**关键改进**：
- 通用的错误处理原则，不涉及具体工具
- 明确的重试策略
- 强调向用户反馈失败原因

### 4. xuanji.yaml

**改进前**：
- 包含硬编码的 task 工具使用规则
- 包含硬编码的工具选择指南
- 包含硬编码的错误场景示例

**改进后**：
- 删除了所有硬编码的工具使用规则
- 保留了通用的错误处理原则和示例
- 不涉及具体工具的使用细节

## 设计优势

### 1. 关注点分离
- **工具 description**：定义工具的使用方法
- **主 agent prompt**：定义通用的行为规范
- **错误信息**：提供具体的修正建议

### 2. 易于维护
- 工具的使用规则变化时，只需修改工具的 description
- 不需要修改主 agent 的 prompt
- 不需要修改其他工具

### 3. 更好的可扩展性
- 添加新工具时，只需定义工具的 description
- 主 agent 自动通过工具的 description 学习如何使用
- 不需要修改主 agent 的 prompt

### 4. 更清晰的职责
- 工具负责说明"如何使用我"
- 主 agent 负责"如何处理错误"和"如何协调工具"
- 错误信息负责"如何修正调用"

## 示例对比

### 改进前的流程

```
用户：分析《出师表》
主 agent：[查看 xuanji.yaml 中的硬编码规则]
主 agent：调用 task，使用 general-purpose
SubAgentFactory：错误，agent 不存在
主 agent：[直接终止，没有重试]
```

### 改进后的流程

```
用户：分析《出师表》
主 agent：[查看 match_agent 的 description]
主 agent：调用 match_agent
match_agent：返回最高分数 0.3（无合适的预置 agent）
主 agent：[查看 task 的 description]
主 agent：根据 description 的指导，创建临时 agent
主 agent：调用 task，提供 system_prompt 和 tools
SubAgentFactory：成功创建临时 agent
```

或者如果第一次失败：

```
主 agent：调用 task，使用 general-purpose
SubAgentFactory：错误，缺少 system_prompt
主 agent：[阅读错误信息]
主 agent：[根据错误提示调整参数]
主 agent：重试 task，提供 system_prompt 和 tools
SubAgentFactory：成功创建临时 agent
```

## 总结

通过这次改进：

1. **工具更加自描述**：每个工具清晰地说明如何使用
2. **主 agent 更加通用**：不依赖硬编码的工具使用规则
3. **错误处理更加智能**：根据错误信息自动重试
4. **系统更加可维护**：工具变化不影响主 agent

这种设计符合"关注点分离"和"单一职责"原则，使系统更加灵活和可扩展。
