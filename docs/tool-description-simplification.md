# 工具 Description 精简原则

## 核心原则

**少而精**：不要让过多的信息误导 LLM，导致 LLM 丢失重点。
- Prompt 内容足够少
- 表达足够精确
- 示例足够通用

## 精简前后对比

### TaskTool

**精简前**（~120 行）：
- 详细的章节结构
- 多个完整示例
- 详细的错误说明
- 工具选择指南

**精简后**（~15 行）：
```
委派任务给子 agent 独立执行。

使用流程：
1. 先调用 match_agent 查找合适的 agent
2. 分数 >= 0.5：使用预置 agent
   分数 < 0.5：创建临时 agent（需 system_prompt + tools）

示例：
task({ description: "...", subagent_type: "coder" })

task({
  description: "...",
  subagent_type: "custom-name",
  system_prompt: "你是...专家，擅长...",
  tools: ["read_file", "grep"]
})
```

**关键改进**：
- 删除冗长的章节标题
- 删除详细的步骤说明
- 删除多个示例，只保留核心示例
- 使用通用占位符（"..."）而不是具体内容
- 保留最关键的决策点（0.5 分数线）

### MatchAgentTool

**精简前**（~100 行）：
- 详细的用途说明
- 三步流程
- 多个完整示例
- 参数说明

**精简后**（~7 行）：
```
查找最合适的预置 agent。在调用 task 前必须先调用此工具。

返回：top 3 推荐 + 匹配分数（0-1）

决策：
- 分数 >= 0.5：使用推荐的预置 agent
- 分数 < 0.5：需创建临时 agent
```

**关键改进**：
- 一句话说明用途
- 一句话说明返回值
- 两行说明决策规则
- 删除所有示例（LLM 可以从 task 的示例中推断）

## 精简规则

### 1. 一句话说明用途
```
✅ 委派任务给子 agent 独立执行。
❌ 将复杂任务委派给专门的子 agent 独立执行。子 agent 有独立的上下文，不共享父 agent 的对话历史。
```

### 2. 核心流程用数字列表
```
✅ 使用流程：
   1. 先调用 match_agent
   2. 分数 >= 0.5：使用预置 agent

❌ ## 使用流程（必须遵循）
   ### 步骤 1：查找合适的 agent
   首先调用 match_agent 查找最合适的预置 agent：
```

### 3. 示例使用通用占位符
```
✅ task({ description: "...", subagent_type: "coder" })

❌ task({
     description: "分析 src/auth.ts 的代码质量，检查潜在问题",
     subagent_type: "coder"
   })
```

### 4. 删除冗余说明
```
✅ 分数 >= 0.5：使用预置 agent

❌ 如果最高分数 >= 0.5（50%）
   说明找到了合适的预置 agent，可以直接使用：
   使用推荐的预置 agent
```

### 5. 删除章节标题
```
✅ 使用流程：
   1. ...
   2. ...

❌ ## 使用流程
   ### 步骤 1：...
   ### 步骤 2：...
```

### 6. 删除重复信息
```
✅ 在调用 task 前必须先调用此工具。

❌ ## 何时使用
   在调用 task 或 agent_team 工具之前，必须先调用此工具查找合适的 agent。
   这确保你使用专门的 agent（coder, explore, test-writer 等），而不是通用的。
```

## 信息层次

### 第一层：核心信息（必须保留）
- 工具用途（一句话）
- 使用流程（关键步骤）
- 决策点（如 0.5 分数线）
- 核心示例（通用占位符）

### 第二层：补充信息（可以删除）
- 详细的步骤说明
- 多个示例
- 错误示例
- 注意事项

### 第三层：背景信息（应该删除）
- 为什么要这样做
- 工具的实现原理
- 历史背景

## 效果对比

### 精简前
- TaskTool: ~120 行
- MatchAgentTool: ~100 行
- 总计: ~220 行

### 精简后
- TaskTool: ~15 行
- MatchAgentTool: ~7 行
- 总计: ~22 行

**减少了 90%，但保留了所有核心信息。**

## 适用范围

这个精简原则应该应用到所有工具的 description：
- ✅ TaskTool
- ✅ MatchAgentTool
- ⏳ 其他工具（待优化）

## 总结

精简的目标是：
1. **减少认知负担**：LLM 更容易抓住重点
2. **提高准确性**：减少误导信息
3. **保持完整性**：核心信息不丢失
4. **提高通用性**：使用通用示例而不是具体案例

记住：**少即是多，精确胜于详细。**
