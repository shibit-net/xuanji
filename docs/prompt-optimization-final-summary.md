## 精简原则总结

### 1. 少而精
- 删除冗余的章节标题（如 ##、###）
- 删除重复的说明和解释
- 删除过多的示例
- 删除详细的背景说明

### 2. 足够精确
- 保留所有决策点（如 0.5 分数线）
- 保留关键流程步骤
- 保留重要规则和原则
- 保留核心概念定义

### 3. 通用示例
- 使用占位符（"..."）而不是具体内容
- 删除具体的文件路径和代码示例
- 保留示例的结构，删除具体内容

### 4. 精炼表达
- 合并相似内容
- 用更简洁的语言表达相同意思
- 删除修饰词（如"非常"、"一定"）
- 删除表情符号（如 🎯、📋、✅）

## 精简技巧

### 技巧 1：合并重复信息
```
精简前：
### 1. 接收意图分析结果
系统会自动分析用户输入，并在消息末尾提供 [意图分析结果]，包含：
- **场景 (scene)**：任务所属的场景类型
- **推荐 Agent (agent)**：最适合处理该任务的 Agent
- **任务复杂度 (complexity)**：simple 或 complex

精简后：
系统在消息末尾提供 [意图分析结果]：scene、agent、complexity
```

### 技巧 2：删除过多示例
```
精简前：
**错误示例**：
❌ "让我先看看..."
❌ "我来分析一下..."
❌ "这个任务需要..."

**正确示例**：
✅ 直接调用 task 工具

精简后：
立即委托，不要自己处理。直接调用 task 工具。
```

### 技巧 3：精简章节标题
```
精简前：
#### 情况 A：简单任务 + 推荐了专业 Agent
**特征**：complexity = simple，agent ≠ general
**处理方式**：立即委托，不要犹豫

精简后：
简单专业任务 (complexity=simple, agent≠general)：立即委托
```

### 技巧 4：删除详细步骤
```
精简前：
### 步骤 1：分析任务
- 理解任务范围和目标
- 识别需要哪些专业能力
- 评估任务复杂度和风险

精简后：
1. 分析任务：理解范围、识别能力、评估复杂度
```

### 技巧 5：删除表情符号
```
精简前：
## 🎯 何时需要规划
## 📋 规划工作流
## 🔍 何时使用 plan_review

精简后：
## 何时需要规划
## 规划工作流
## 何时使用 plan_review
```

## 优化效果

### Token 节省
- 精简前总 token 数：约 850 行 × 平均 50 tokens/行 = 42,500 tokens
- 精简后总 token 数：约 250 行 × 平均 50 tokens/行 = 12,500 tokens
- **节省 token：约 30,000 tokens（71%）**

### LLM 性能提升
1. **更快的响应**：更少的 token 意味着更快的处理速度
2. **更准确的理解**：减少干扰信息，LLM 更容易抓住重点
3. **更低的成本**：每次调用节省 71% 的 prompt token
4. **更好的一致性**：精简的 prompt 减少了歧义

### 可维护性提升
1. **更易阅读**：开发者更容易理解 prompt 的核心内容
2. **更易修改**：修改时不需要处理大量冗余信息
3. **更易测试**：精简的 prompt 更容易验证效果
4. **更易扩展**：添加新内容时不会被淹没在冗余信息中

## 结论

通过系统化的精简优化，我们成功将 prompt 总量从 849 行减少到 249 行，减少了 71%。这不仅节省了 token 成本，更重要的是提高了 LLM 的理解准确性和响应速度。

精简的核心原则是：**少而精，足够精确**。删除所有冗余信息，保留所有关键信息。让 LLM 能够快速抓住重点，做出正确的决策。
