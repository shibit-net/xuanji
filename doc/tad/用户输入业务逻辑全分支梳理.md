# 璇玑 (xuanji) 用户输入后完整业务逻辑分支梳理

## 一、整体架构概览

```
用户输入 (InputArea)
  │
  ▼
Electron 渲染进程 → IPC → 主进程 → ChildMessageChannel → agent-bridge 子进程
  │
  ▼
handleUserAction() → IntentRouter.route() → SessionStateMachine.transition() → ChatSession.run() → AgentLoop.run()
```

核心进程模型：**agent-bridge 是独立 Node.js 子进程**，通过 IPC MessageBus 与 Electron 主进程通信，主进程再转发给渲染进程。

---

## 二、用户输入后的第一层：意图路由 (IntentRouter)

发生在 `agent-bridge.ts` 的 `handleUserAction()` 中，**三级降级路由**：

### L1 — LLM 意图分析 (SceneClassifier)

- 调用 `scene-classifier` agent（使用 `qwen2.5-1.5b-q4` 小模型，temperature=0.3，maxTokens=256）
- 输出 JSON：`{ scene, agent, complexity, confidence }`
- **失败条件**：LLM 不可用 / 超时（15s） / JSON 解析失败 / 返回的 agent 不在 Registry 中 → **降级到 L2**
- **成功条件**：agent 在 Registry 中存在 → 直接返回，method=`llm`

### L2 — 关键词+能力匹配 (EmbeddingMatcher)

- 纯本地计算，不依赖任何外部服务
- 对每个启用的 agent（排除 system/internal/mainAgent）计算：
  - `keywordScore`：用户消息分词后与 agent 的 id+name+description+capabilities 做词重叠
  - `capabilityScore`：任务描述词与能力描述词做双向重叠匹配
  - `totalScore = keywordScore × 0.5 + capabilityScore × 0.5`
- **生效条件**：`topMatch.score >= 0.5` 且 agent 在 Registry 中存在
- **失败** → 降级到 L3

### L3 — 默认兜底

- 永远返回 `xuanji`（主 agent），confidence=1.0，method=`default`

### 路由结果后的动作

```typescript
// agent-bridge.ts handleUserAction()
routedAgentId = route.agentId;
await session.switchForegroundAgent(route.agentId, route.scene, route.complexity);
// 发送路由结果 IPC → 前端展示 IntentDialog
channel.send('agent:intent-route', { agentId, confidence, method, scene, complexity });
channel.send('agent:switch-foreground', { agentId, name });
```

`switchForegroundAgent()` 做了什么：

1. 从 AgentRegistry 获取 agent 配置
2. 用 `LayeredPromptBuilder` 分层构建 system prompt（L0 基础层 + L1 场景层 + L2 协调层 + agent 自身 prompt）
3. 按 agent 的 tools 列表创建 `FilteredToolRegistry`
4. 调用 `agentLoop.applyAgentConfig()` 热替换 provider / systemPrompt / toolRegistry / model / maxIterations / temperature

## 三、第二层：Session 状态机 (SessionStateMachine)

4 个状态，6 种事件：

```
         USER_MESSAGE                    USER_MESSAGE
  idle ──────────────► executing ─────────────────────► (排队)
   ▲                    │    │
   │                    │    │ AGENT_TEXT_STARTED
   │                    │    ▼
   │                    │  outputting
   │                    │    │
   │                    │    │ USER_MESSAGE → 排队 + 设置 abort 标记
   │                    │    │ USER_INTERRUPT → ABORT_AGENT
   │                    │    │
   │     AGENT_COMPLETED│    │ AGENT_COMPLETED
   │     (无排队无异步)   │    │
   │   ◄────────────────┘    │
   │                          │ AGENT_COMPLETED (有异步任务未完成)
   │                          ▼
   │                     waiting_async
   │                          │
   │                          │ ASYNC_TASK_COMPLETED (所有完成, 无排队)
   │                          ▼
   └────────────────────── idle
```

### 状态转移表

| 当前状态 | 事件 | 动作 | 下一状态 |
|---------|------|------|---------|
| **idle** | USER_MESSAGE | RUN_AGENT | executing |
| **idle** | USER_INTERRUPT | QUEUE_ONLY (排队消息) | idle |
| **executing** | USER_MESSAGE | QUEUE_ONLY (排队) | executing |
| **executing** | USER_INTERRUPT | ABORT_AGENT + 排队新消息 | executing |
| **executing** | AGENT_TEXT_STARTED | NOOP | outputting |
| **executing** | AGENT_COMPLETED | 有排队→RUN_AGENT / 有异步→waiting_async / 无→idle | 各异 |
| **outputting** | USER_MESSAGE | QUEUE_ONLY + 设abort标记 | outputting |
| **outputting** | USER_INTERRUPT | ABORT_AGENT + 排队 | outputting |
| **outputting** | AGENT_COMPLETED | (同executing的AGENT_COMPLETED) | 各异 |
| **waiting_async** | USER_MESSAGE | RUN_AGENT(立即) | executing |
| **waiting_async** | USER_INTERRUPT | QUEUE_ONLY | waiting_async |
| **waiting_async** | ASYNC_TASK_COMPLETED | 全部完成+无排队→idle / 全部完成+有排队→RUN_AGENT / 还有未完成→RUN_AUTO_SUMMARIZE | 各异 |

## 四、第三层：AgentLoop ReAct 循环

`AgentLoop.run()` 核心流程：

```
1. 发射 AGENT_STARTED → SessionStateMachine.transition(AGENT_STARTED)
2. 用户消息入 ContextManager
3. while (迭代 < 100 且 未abort):
   ├── Budget 压缩检查 (历史 > 10条 → 红色压缩 / 黄色压缩)
   ├── StreamPipeline.execute() → 调用 LLM
   ├── checkShouldStop() → 检查点 A (abort? 有待处理输入且尚未开始输出?)
   ├── 无 tool call → 跳出
   ├── ToolGateway.executeBatch() → 并行/串行执行工具
   ├── checkShouldStop() → 检查点 B
   ├── tool results 入 ContextManager
   ├── 委派完成检测 (task/agent_team 工具无错误 → 注入 delegation-complete)
   └── 卡住检测:
       ├── 同一工具连续失败 >= 2 → stuck-detect-tool-fail 警告
       ├── 同一文件连续读 >= 3 → stuck-detect-same-file 警告
       └── 连续 2+ 次相同输出 → 跳出循环
4. finally: AGENT_COMPLETED → SessionStateMachine.transition(AGENT_COMPLETED)
```

**中断检查 (InterruptChecker)**：

- AgentLoop 每轮循环在 StreamPipeline 中注入 `InterruptChecker`
- Phase 2 路径：`SessionStateMachine` 实现了 `InterruptChecker` 接口
- `shouldStop()` = `_abortRequested || (pendingMessages.length > 0 && !_textOutputStarted)`
- 含义：**如果用户正在打字（尚未开始输出），则打断当前执行并处理新输入**
- 如果已经开始输出，则只排队不打断

## 五、多 Agent 场景全景

### 5.1 前台 Agent（Foreground Agent）

用户每次输入，经过 IntentRouter 路由到目标 agent，通过 `switchForegroundAgent()` 热切换。

**关键行为**：

- 只有一个前台 agent 在运行
- 所有用户可见的文本输出、工具调用都来自前台 agent
- `AgentStateMachine` 中有 `foregroundAgentId` 字段追踪
- agent 完成一轮后 → `FOREGROUND_COMPLETE` → 状态回 `pending`（等待下一轮）

### 5.2 后台 Task（task 工具）

前台 agent 通过调用 `task` 工具创建后台子 agent：

```
前台 agent 调用 task({ subagent_type: "coder", description: "..." })
  │
  ▼
AgentFactory.createAndRun() → 创建子 AgentLoop → 后台执行
  │
  ▼
TaskOrchestrator.startTask()
  ├── AsyncTaskStateMachine: creating → running
  ├── 执行完毕后: completed / failed / cancelled
  │     │
  │     ▼
  │   TaskCompletionHandler.handleCompletion()
  │     ├── 注入 async-task-completion 到 system prompt 后缀
  │     │
  │     └── 若主 agent 空闲 (isRunning=false)：
  │         autoSummarize() → onRun('[系统通知]...') → 主 agent 自动汇总汇报
  │
  └── 若主 agent 正在运行：
      完成通知排队，主 agent 本轮结束后 checkAndAutoSummarize()
```

**并发控制**：`maxConcurrent = 3`，超过上限返回错误。

**超时控制**：`maxLifetimeMs = 4 小时`。

### 5.3 后台 Team（agent_team 工具）

前台 agent 通过调用 `agent_team` 工具创建团队。这是最复杂的多 agent 场景。

```
agent_team({ name: "代码审查组", strategy: "debate", members: [...] })
  │
  ▼
TeamManager.createTeam() → TeamManager.execute()
  │
  ├── 预生成所有成员的 subAgentId (成员 ID 一致性)
  ├── 发射 HOOK_TEAM_START (IPC → 前端展示团队节点)
  ├── 执行策略:
  │   ├── sequential  (串行)
  │   ├── parallel    (并行，最多 3 并发 + git worktree 隔离)
  │   ├── hierarchical (层级：Leader 分解 → task 工具分配)
  │   ├── debate      (辩论：正方/反方/裁判 + 多轮)
  │   └── pipeline    (流水线：文件传递)
  │
  ├── 每个成员执行:
  │   ├── 发射 HOOK_TEAM_MEMBER_START (前端展示成员节点)
  │   ├── AgentFactory.createAndRun() → 子 AgentLoop
  │   ├── 发射 HOOK_TEAM_MEMBER_END
  │   ├── 保存 checkpoint (memberId.json)
  │   └── 失败重试 1 次
  │
  └── 结果聚合 → 发射 HOOK_TEAM_END
```

#### 五种团队策略详解

| 策略 | 执行方式 | 结果聚合 | 超时公式 |
|------|---------|---------|---------|
| **sequential** | 按 priority 排序，依次执行，任一失败则停止 | 返回最后一个成员结果 | 300s + 100s×(N-1) |
| **parallel** | 并发窗口=3，为每个成员创建独立 git worktree | 合并所有成员结果 | 600s + 200s×(N-1) |
| **hierarchical** | Leader 通过 task 工具分配子任务给 Worker | Leader 输出 + Worker 结果 | 600s + 300s×(N-1) |
| **debate** | Judge预读 → 多轮辩论 → 论点新颖度检测 → 早停 → Judge裁决 → 自动写文件 | 辩论记录 + 裁决 + 文件 | 1800s × 轮次因子 |
| **pipeline** | 成员定义顺序执行，阶段间通过文件传递数据 | 最后一个阶段输出 | 300s + 150s×(N-1) |

#### Debate 策略的特殊逻辑

- **Judge 预读模式**：Judge 先读取关键文件输出「事实摘要」，正反方引用摘要而非重复读文件
- **论点新颖度检测**：Jaccard 相似度对比相邻轮次，新颖度 < 阈值时早期收敛
  - `consecutiveLowNoveltyRounds >= 2` → 触发提前裁决
  - 新颖度 < 10% → 立即裁决
  - 第 3 轮后新颖度 < 30% → 裁决
- **共识检测**：所有正反方输出包含 agree/consensus/一致/同意 关键词 → 结束辩论
- **自动写文件**：辩论结束后自动将共识写入目标文件

#### Checkpoint 恢复机制

- 团队执行时，每个成员完成后保存 checkpoint 到 `.xuanji/checkpoints/<teamName>/<memberId>.json`
- 下次同 teamName 的团队执行时，`loadCheckpoints()` 恢复成功成员的结果
- 恢复的成员直接跳过执行，但发射完整的 TeamMemberStart/End Hook 保证前端一致性
- 全部成功后清除所有 checkpoint

## 六、用户中途加入（Mid-stream Interrupt）

这是最复杂的交互场景。核心入口是 `SessionStateMachine.transition()`。

### 场景 A：Agent 正在 thinking/executing（尚未开始输出文本）

```
用户输入新消息
  → SessionStateMachine.handleInExecuting(USER_MESSAGE)
  → pendingMessages.push(message)
  → 返回 QUEUE_ONLY
  → 消息排队，AgentLoop 的下一个 checkShouldStop() 会检测到
  → shouldStop() = true (pendingMessages.length > 0 && !_textOutputStarted)
  → AgentLoop 本轮结束后跳出
  → ChatSession.run() 的 handleAgentCompleted() 检测到 pendingMessages
  → consumePendingMessages() 合并所有排队消息用 \n 拼接
  → 递归调用 run(合并消息) — 一轮处理所有排队
```

### 场景 B：Agent 正在 outputting（已经给用户打字）

```
用户输入新消息
  → SessionStateMachine.handleInOutputting(USER_MESSAGE)
  → pendingMessages.push(message)
  → _abortRequested = true
  → 返回 QUEUE_ONLY
  → AgentLoop 检测到 abort → 停止 LLM 流 + 停止工具执行
  → 本轮结束后处理排队消息
```

### 场景 C：用户点击中断按钮（INTERRUPT，无新输入）

```
  → executing 状态 → ABORT_AGENT
  → outputting 状态 → ABORT_AGENT
  → AgentLoop.requestAbort() / stop()
```

### 场景 D：用户点击中断 + 带新输入

```
  → pendingMessages 头部插入新输入（unshift，优先处理）
  → ABORT_AGENT
  → 中断完成后立刻 drainPendingQueue → 执行新输入
```

### 排队消息的批处理 (drainPendingQueue)

旧路径中 `drainPendingQueue()` 循环逐个消费；Phase 2 新路径中 `consumePendingMessages()` 一次性合并所有排队消息，用 `\n` 拼接成一个消息，**一轮 run() 处理所有**，避免多次 AgentLoop 重启。

## 七、数据流全景

### 7.1 核心事件流

```
用户输入
  → IPC: 'user-action' { type: 'SEND_MESSAGE', message }
  → agent-bridge: handleUserAction()
      ├── IntentRouter.route() → 三级路由
      │     ├── IPC: 'agent:intent-route:start'
      │     ├── IPC: 'agent:intent-route:progress' (L1/L2/L3 逐级)
      │     └── IPC: 'agent:intent-route' (最终结果)
      ├── switchForegroundAgent(agentId, scene, complexity)
      │     └── IPC: 'agent:switch-foreground'
      └── session.userAction(data)
            └── SessionStateMachine.transition(USER_MESSAGE)
                  └── ChatSession.run(message)

AgentLoop.run():
  ├── EventBus: AGENT_STARTED → IPC: 'agent:started'
  ├── StreamPipeline 逐 token:
  │     ├── EventBus: AGENT_TEXT_DELTA → IPC: 'agent:text'
  │     └── EventBus: AGENT_THINKING_DELTA → IPC: 'agent:thinking'
  ├── 工具执行:
  │     ├── EventBus: AGENT_TOOL_START → IPC: 'agent:tool-start'
  │     └── EventBus: AGENT_TOOL_END → IPC: 'agent:tool-end'
  └── EventBus: AGENT_COMPLETED → IPC: 'agent:end'

子 Agent/Team:
  ├── EventBus: HOOK_SUBAGENT_START → IPC: 'agent:subagent-start'
  ├── EventBus: HOOK_SUBAGENT_END → IPC: 'agent:subagent-end'
  ├── EventBus: HOOK_TEAM_START → IPC: 'agent:team-start'
  ├── EventBus: HOOK_TEAM_MEMBER_START → IPC: 'agent:team-member-start'
  ├── EventBus: HOOK_TEAM_MEMBER_END → IPC: 'agent:team-member-end'
  └── EventBus: HOOK_TEAM_END → IPC: 'agent:team-end'

异步任务完成（两个发送路径，IPC 事件名相同）:
  ├── 旧路径 (registerHookEventBridge):
  │     EventBus: ASYNC_TASK_COMPLETED → IPC: 'agent:task-completed'
  │     EventBus: ASYNC_TASK_FAILED    → IPC: 'agent:task-failed'
  └── 新路径 (EventForwarder):
        AsyncTaskStateMachine.onTaskStateChanged → IPC: 'agent:task-completed' / 'agent:task-failed'

其他:
  ├── Skill 生命周期: HOOK_SKILL_START → IPC: 'agent:skill-start'
  │                   HOOK_SKILL_END   → IPC: 'agent:skill-end'
  ├── 子 Agent 引用:  TaskCompletionHandler → IPC: 'agent:citation-data'
  ├── 项目信息:       agent-bridge initWorkspace() → IPC: 'project:info'
  └── 错误:          EventBus: AGENT_ERROR → IPC: 'agent:error' (当前无前端消费)
```

### 7.2 IPC → 前端 Store 映射 (EventAdapter)

```
agent:started          → ConversationStore.onAgentStarted() + messageStore.startStreaming()
agent:text             → AgentStateMachine.TEXT_DELTA + messageStore.appendStreamingText()
agent:thinking         → AgentStateMachine.THINKING_DELTA
agent:tool-start       → AgentStateMachine.TOOL_START
agent:tool-end         → AgentStateMachine.TOOL_END
agent:end              → ① FOREGROUND_COMPLETE (前台回 pending)
                          → ② CLEANUP_COMPLETED_TASKS (清理后台终态，依赖前台已回 pending)
                          → ③ CLEAR_QUEUED_MESSAGE (清除排队计数)
                          → ConversationStore.onAgentCompleted() + messageStore.finishStreaming()
agent:subagent-start   → AgentStateMachine.AGENT_CREATED + AsyncTaskStore.TASK_CREATED
agent:subagent-end     → AgentStateMachine.SUBAGENT_END + AsyncTaskStore.TASK_COMPLETED
agent:team-start       → AgentStateMachine.AGENT_CREATED + AsyncTaskStore.TASK_CREATED
agent:team-member-start → AgentStateMachine.AGENT_CREATED + AsyncTaskStore.MEMBER_STATE_CHANGED
agent:team-member-end   → AgentStateMachine.SUBAGENT_END + AsyncTaskStore.MEMBER_STATE_CHANGED
agent:team-end         → 批量 CLEANUP 所有后代 + AsyncTaskStore.TASK_COMPLETED
agent:intent-route:start    → IntentRoutingStore.ROUTE_START (idle → analyzing)
agent:intent-route:progress → IntentRoutingStore.ROUTE_STAGE (逐级追加)
agent:intent-route          → IntentRoutingStore.ROUTE_COMPLETE + ConversationStore.setRoutingInfo
agent:auto-summarize   → AgentStateMachine.AUTO_SUMMARIZE_START + AsyncTaskStore.TASK_CLEARED
agent:conversation-state → ConversationStore.setConversationState()
agent:message-queued   → AgentStateMachine.QUEUED_MESSAGE（queuedMessageCount++）
agent:queue-consumed   → AgentStateMachine.CLEAR_QUEUED_MESSAGE
agent:citation-data    → CitationStore.addCitation (子 agent 输出引用)
agent:skill-start      → ConversationStore.setActiveSkill
agent:skill-end        → ConversationStore.setActiveSkill(null)
agent:error            → 当前无前端 Store 消费（仅用于日志）
session:init-*         → SessionInitStore (uninitialized → initializing → ready/failed)
```

## 八、任务结束的各类路径

### 8.1 正常结束

- AgentLoop ReAct 循环中无 tool call 或 stopReason=end_turn → 跳出循环
- → `AGENT_COMPLETED` → `SessionStateMachine.handleAgentCompleted()`
- 检查是否有排队消息 → 有则 `RUN_AGENT`
- 检查是否有未完成的异步任务 → 有则 `waiting_async`
- 都没有 → `idle`

### 8.2 异步任务完成后的汇总汇报

- `TaskCompletionHandler.autoSummarize()` 被触发
- 向 ContextManager 注入 `async-task-completion` 后缀
- 调用 `onRun('[系统通知] 有一个后台任务刚完成...')` → `ChatSession.run()`
- 主 agent 自动汇总结果给用户
- 汇总完成后如果有更多 pending → 递归触发下一个

### 8.3 用户主动中断

- `USER_INTERRUPT` → `ABORT_AGENT` → `agentLoop.requestAbort()` / `agentLoop.stop()`
- 中断 LLM 流 + 正在执行中的工具
- `AGENT_COMPLETED` → 正常走 handleAgentCompleted

### 8.4 卡住检测自动终止

- 连续 2+ 次相同输出文本 → 注入终止提示，跳出循环
- 同一工具连续失败 2+ 次 → stuck-detect-tool-fail 注入 system prompt
- 同一文件连续读 3+ 次 → stuck-detect-same-file 注入 system prompt

### 8.5 异常终止

- 未捕获异常 / Promise rejection → `process.on('uncaughtException')` / `unhandledRejection`
- 发送 `agent:error` + `agent:end` IPC → 前端恢复 idle
- 调用 `session.stop()` → 进程退出（exit code 1）

### 8.6 超时终止

- **Agent 级别**：`HARD_MAX_ITERATIONS = 100`
- **Task 级别**：`maxLifetimeMs = 4 小时`
- **Team 级别**：策略特定公式计算（30分钟到90分钟不等）
- **成员级别**：策略特定公式计算（30秒到180秒不等）
- 超时 → `AbortController.abort()` → 子 AgentLoop 抛出 AbortError

## 九、前端状态机 (AgentStateMachine) — 9 种状态

```
pending ──► thinking ──► executing ──► writing ──► success
                                                    failed
                                                    cancelled
                                                    reporting ──► cleared
```

**终态屏障**：一旦 agent 进入 success/failed/cancelled/cleared，除 CLEANUP 外的事件全部忽略。

**主要状态转换**：

- `THINKING_DELTA` → pending → thinking
- `TOOL_START` → thinking → executing
- `TEXT_DELTA` → thinking/executing → writing
- `SUBAGENT_END` (success) → writing → reporting
- `SUBAGENT_END` (failed) → failed
- `AUTO_SUMMARIZE_START` → cleared
- `CLEANUP` → cleared (team 结束、任务清除)

**排队计数**：`queuedMessageCount` 通过 `QUEUED_MESSAGE` / `CLEAR_QUEUED_MESSAGE` 事件维护，前端展示排队数量。

## 十、总结：关键业务分支决策树

```
用户输入消息
  │
  ├─ Session 未初始化? → SessionInitStore: uninitialized → 触发 init → 阻塞等待 ready
  │
  ├─ 三级意图路由
  │   ├─ L1 LLM 分析成功 → 路由到目标 agent
  │   ├─ L1 失败 → L2 关键词匹配成功 → 路由到匹配 agent
  │   └─ L2 失败 → L3 默认 → xuanji
  │
  ├─ Session 状态检查
  │   ├─ idle → 直接切换 agent + 启动 AgentLoop
  │   ├─ executing → 排队（AgentLoop 下个检查点处理）
  │   ├─ outputting → 排队 + 设置 abort 标记（输出完成后处理）
  │   └─ waiting_async → 立即 RUN_AGENT（打断等待）
  │
  ├─ AgentLoop ReAct 循环
  │   ├─ 无 tool call → 结束 → 汇报结果
  │   ├─ task 工具 → 创建后台子 agent → 注入 delegation-complete → 继续
  │   ├─ agent_team 工具 → 创建团队（5 种策略） → 注入 delegation-complete → 继续
  │   └─ 普通工具 → 执行 → 继续
  │
  ├─ 本轮结束后
  │   ├─ 有排队消息 → 合并后 RUN_AGENT
  │   ├─ 有未完成异步任务 → waiting_async
  │   └─ 都没有 → idle
  │
  └─ 后台任务完成后
      ├─ 主 agent 空闲 → TaskCompletionHandler.autoSummarize() → 自动汇总汇报
      └─ 主 agent 运行中 → 排队，本轮结束后汇总
```
