## 📋 施工总览

### 目标

将当前混乱的状态管理（防御代码 80+ 处、4 种递归树搜索、3 条 IPC 通道、3 层 reentrant guard、50ms 轮询）改造为状态机驱动的干净架构：
- **后端**：~2500 行 → ~1400 行（-44%）
- **前端**：~3300 行 → ~900 行（-73%）
- **IPC 通道**：3 条 → 1 条
- **状态变量**：3 个 → 1 个（派生）

### 时间线

| 阶段 | 用时 | 内容 | 交付物 |
|------|------|------|--------|
|| **Phase 1** | W1 | 新建模块 | 10 个新文件（零风险） |
| **Phase 2** | W2 | 后端改造 | ChatSession、AgentLoop、agent-bridge、main process |
| **Phase 3** | W3 | 前端 IPC + EventAdapter | preload、EventAdapter |
| **Phase 4** | W3-4 | 组件逐一切换 | 13 个组件 |
| **Phase 5** | W4 | 清理旧代码 | 删除 5+ 旧模块 |

### Feature Flag 矩阵

CI 必须覆盖以下 3 种组合：

| Profile | SessionStateMachine | EventForwarder | AgentStateMachine | New IPC | SessionStore |
|---------|:-:|:-:|:-:|:-:|:-:|
| all-off（基线） | OFF | OFF | OFF | OFF | OFF |
| backend-new | ON | ON | OFF | OFF | OFF |
| all-on（目标） | ON | ON | ON | ON | ON |

```typescript
// 重点：handleAgentCompleted 自动消费队列
private handleAgentCompleted(): SessionAction {
  if (this.context.pendingMessages.length > 0) {
    // 有排队消息 → 继续运行（替代 drainPendingQueue + \n join）
    return { type: 'RUN_AGENT', message: batch.join('\n') };
  }
  if (this.context.hasPendingAsyncTasks) {
    this.transitionTo('waiting_async');
    return { type: 'NOOP' };
  }
  this.transitionTo('idle');
  return { type: 'EMIT_SESSION_IDLE' };
}
```

```typescript
    // Tool

    { event: XuanjiEvent.AGENT_TOOL_END,       channel: 'agent:tool-end',        map: this.mapAgent },

    { event: XuanjiEvent.AGENT_FILE_CHANGES, channel: 'agent:citation', map: this.mapSession },
```

## 🛠️ Phase 2：后端改造（W2，中风险）

### 2.1 ChatSession 改造（`src/core/chat/ChatSession.ts`）

**参考**：backend-migration-plan.md 步骤 2

**改动清单**：
1. `_pendingQueue`、`_drainRunning`、`_currentAgentId` — **Phase 2 保留**（feature flag 共存），Phase 5 删除
2. 构造函数中，`StateTracker` 与 `SessionStateMachine` 共存（feature flag 控制走哪条路径）
3. `handleUserInput()` / `interrupt()` / `appendMessage()` — Phase 2 保留，新增 `userAction(action)` 统一入口
4. `run()` 简化：新旧两套 reentrancy guard 共存（flag 控制）
5. `interrupt()` 中的 50ms 轮询 — Phase 2 保留，新路径由 `AgentLoop.onEnd` 回调驱动
6. `drainPendingQueue()` — Phase 2 保留，新路径由 `handleAgentCompleted` 自动消费
7. `agentLoop.setPendingQueue()` — Phase 2 保留（旧路径），新路径注入 `InterruptChecker`
8. `setupAgentLoopCallbacks()` 调用状态机 `transition()` — 新旧两套回调共存

**改造后结构**：
```
ChatSession.userAction(action)
  ├─ IntentRouter.route()     ← Scene 系统未来可替换
  ├─ SessionStateMachine.transition(event)
  │    └─ 返回 SessionAction
  ├─ switch(action.type):
  │    RUN_AGENT           → this.run(msg)
  │    ABORT_AGENT         → agentLoop.requestAbort()
  │    EMIT_SESSION_IDLE   → emitConversationState('idle')
  │    RUN_AUTO_SUMMARIZE  → TaskCompletionHandler
  │    NOOP                → (无操作)
  └─ 返回 ActionResult
```

**并行策略**：feature flag `USE_SESSION_STATE_MACHINE` 控制新旧逻辑切换。默认 `false`。

```typescript
if (process.env.USE_SESSION_STATE_MACHINE === 'true') {
  return this.stateMachine.transition(event); // 新逻辑
} else {
  switch (this.stateTracker.getState()) { ... } // 旧逻辑
}
```

```typescript
class AgentLoop {
  private sessionState: InterruptChecker;
  async run(input: string): Promise<void> {
    while (this.running && this.iteration < this.maxIterations) {
      if (this.sessionState.shouldAbort()) break;
      const response = await this.streamPipeline.execute(currentInput);
      if (this.sessionState.shouldStop()) break;
      const toolResults = await this.toolGateway.executeBatch(response.toolCalls);
      currentInput = this.buildNextInput(response, toolResults);
    }
  }
}
```

### 2.6 EventBus + HookRegistry 边界确认

**参考**：backend-migration-plan.md 步骤 10

**结论**：**不需要合并**。EventBus（生命周期级）与 HookRegistry（工具级拦截）职责不同。
`EventForwarder` 同时监听两者产出的 EventBus 事件即可。

**EventForwarder 需注册的 Hook 相关事件**（与 1.1.2 节的映射表对齐）：
```typescript
private registerHookEvents(): void {
  // HookRegistry 结果事件 → 前端 AgentStateMachine 消费
  this.eventBus.on(XuanjiEvent.HOOK_SUBAGENT_START, (p) => {
    this.safeSend({ type: 'agent:subagent-start', data: this.mapAgent(p) });
  });
  this.eventBus.on(XuanjiEvent.HOOK_SUBAGENT_END, (p) => {
    this.safeSend({ type: 'agent:subagent-end', data: this.mapAgent(p) });
  });
  this.eventBus.on(XuanjiEvent.HOOK_TEAM_START, (p) => {
    this.safeSend({ type: 'agent:team-start', data: this.mapTeam(p) });
  });
  this.eventBus.on(XuanjiEvent.HOOK_TEAM_MEMBER_START, (p) => {
    this.safeSend({ type: 'agent:team-member-start', data: this.mapTeamMember(p) });
  });
  this.eventBus.on(XuanjiEvent.HOOK_TEAM_MEMBER_END, (p) => {
    this.safeSend({ type: 'agent:team-member-end', data: this.mapTeamMember(p) });
  });
  this.eventBus.on(XuanjiEvent.HOOK_TEAM_END, (p) => {
    this.safeSend({ type: 'agent:team-end', data: this.mapTeam(p) });
  });
}
```

> ⚠️ **编码注意**：1.1.2 节 `EventForwarder` 映射表中 `HOOK_SUBAGENT_START` / `HOOK_SUBAGENT_END` / `HOOK_TEAM_*` 等事件的**实际发射源来自 HookRegistry**（通过 HookEventEmitter 触发后，AgentFactory/TeamManager 调用 `eventBus.emit(XuanjiEvent.HOOK_*)`）。EventForwarder 不需要直接引用 HookRegistry，只需要监听对应的 EventBus 事件名即可。编码时请确认 `src/core/events/events.ts` 中 `HOOK_*` 常量的确切命名。
