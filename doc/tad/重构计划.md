## 一、项目概述

### 1.1 重构目标

将璇玑前端+后端的 Agent 状态管理体系从"事件驱动 + 手动同步"模式改造为"状态机驱动"模式，消除防御性补丁代码、统一 IPC 通道、消灭轮询等待。

| 指标 | 当前 | 目标 | 改善 |
|------|------|------|------|
| 后端代码行数 | ~2500 | ~1400 | -44% |
| 前端代码行数 | ~3300 | ~900 | -73% |
| IPC 通道数 | 3 条 | 1 条 | -66% |
| 前端状态变量 | 3 个独立维护 | 1 个派生 | -66% |
| 树搜索实现 | 4 种递归 | 0 种（Map O(1)） | 消除 |
| 终态防御检查 | 7+ 处 | transition 统一 1 处 | -86% |
| 50ms 轮询 | interrupt() 中 | 事件驱动回调 | 消除反模式 |

### 1.2 重构范围

**包含**：
- 后端：新建 SessionStateMachine + EventForwarder + AsyncTaskStateMachine + InterruptChecker
- 后端改造：ChatSession、AgentLoop、agent-bridge、main process IPC、AgentFactory
- 前端：新建 AgentStateMachine + AsyncTaskStore + SessionStore + CitationStore + StoreEventBus + flowBuilder
- 前端新建（Phase 3）：EventAdapter
- 前端改造：10 个组件切换数据源
- 旧代码清理：删除 5 个旧模块 + 简化 3 个文件

**不包含**：
- Scene 系统实现（仅预留接入路径）
- ACP 进程模型变更（不受影响）
- HookRegistry 重写（不受影响）
- Handoff 系统变更（不受影响）
- 新增业务功能

### 1.3 成功定义

- [ ] 所有 7 个核心用户场景（施工文档 Checklist）端到端通过
- [ ] CI 3 种 feature flag profile 全绿
- [ ] 新旧系统并行运行 1 周，事件日志无差异
- [ ] 代码行数降幅达到目标（后端 -44%、前端 -73%）
- [ ] 无 P0/P1 回归 bug

## 四、Feature Flag 策略

### 4.1 Flag 定义

| Flag | 默认值 | 控制范围 | 引入 Phase |
|------|--------|---------|-----------|
| `USE_SESSION_STATE_MACHINE` | `false` | 后端 ChatSession 走新旧路径 | Phase 2 |
| `USE_EVENT_FORWARDER` | `false` | EventForwarder 替代 registerHookEventBridge | Phase 2 |
| `USE_AGENT_STATE_MACHINE` | `false` | 前端 EventAdapter 路由到新 Store | Phase 3 |
| `USE_NEW_IPC` | `false` | 前端 InputArea 使用 agentUserAction | Phase 4 |
| `USE_SESSION_STORE` | `false` | 前端组件读取 SessionStore 替代 messageStore.status | Phase 4 |

### 4.2 Flag 切换时序

```
W2: USE_SESSION_STATE_MACHINE=true, USE_EVENT_FORWARDER=true  (backend-new profile)
W3: + USE_AGENT_STATE_MACHINE=true
W4: + USE_NEW_IPC=true, USE_SESSION_STORE=true                 (all-on profile)
W5: 默认 true，删除旧代码路径
```

### 4.3 CI 矩阵

CI 必须同时运行以下 3 种 profile：

| Profile | SESSION | FORWARDER | AGENT | IPC | STORE | 用途 |
|---------|:-:|:-:|:-:|:-:|:-:|------|
| all-off | OFF | OFF | OFF | OFF | OFF | 基线，确保不改坏现有功能 |
| backend-new | ON | ON | OFF | OFF | OFF | 验证后端状态机 + EventForwarder |
| all-on | ON | ON | ON | ON | ON | 验证目标架构 |

### 5.2 回滚策略

每个 Phase 有独立回滚路径：

| Phase | 回滚方式 | 回滚时间 |
|-------|---------|---------|
| Phase 1 | 无需回滚（新文件独立，不影响运行） | — |
| Phase 2 | 关闭 `USE_SESSION_STATE_MACHINE` + `USE_EVENT_FORWARDER` | < 1 分钟（环境变量） |
| Phase 3 | 关闭 `USE_AGENT_STATE_MACHINE`，保留旧 EventBridge | < 1 分钟（环境变量） |
| Phase 4 | 关闭 `USE_NEW_IPC` + `USE_SESSION_STORE`，组件仍读旧 Store | < 1 分钟（环境变量） |
| Phase 5 | `git revert` 清理 commit，恢复被删文件 | < 5 分钟 |

**全量回滚**：将全部 5 个 flag 设为 `false`，系统回到改造前状态。任何时刻可用。

### 5.3 止损条件

出现以下任一情况时暂停推进，回退到上一 Phase 的稳定状态：

- P0 问题（Agent 不响应 / 事件丢失导致 UI 卡死 / 消息丢失）
- 连续 2 个 P1 问题未在 24h 内修复
- CI all-off profile 失败（说明改造破坏了基线）
- 7 个核心场景中任一失败

### 5.4 Phase 依赖关系

```
Phase 1 [新建模块]
  └─ 无外部依赖，可独立进行

Phase 2 [后端改造]
  └─ 依赖: Phase 1 的 SessionStateMachine / EventForwarder / AsyncTaskStateMachine / InterruptChecker 就绪
  └─ 注意: 前端 Phase 3 依赖 Phase 2 的 IPC 协议稳定（agent:user-action 通道 + EventForwarder 事件名）

Phase 3 [前端 IPC]
  └─ 依赖: Phase 2 的 IPC 协议就绪（事件名确定后才能写 EventAdapter）
  └─ 注意: 可在 Phase 2 门禁通过后立即启动，不等待 Phase 4

Phase 4 [组件切换]
  └─ 依赖: Phase 3 的 EventAdapter + 新 Store 就绪
  └─ 并行: 低风险组件（StatusBar/BackgroundTaskIndicator）可在 Phase 3 Day 5 提前开始

Phase 5 [清理]
  └─ 依赖: Phase 4 门禁通过（all-on profile 验证通过后才能安全删除旧代码）
```

### 5.5 范围变更流程

发现施工文档未覆盖的改造项时：

1. 在对应 Review 文档中记录变更（记录时间 + 影响面）
2. 评估对工期的影响（增加多少天）
3. 更新施工文档对应章节
4. PM 确认是否纳入当前重构或推迟到后续迭代
5. 如果影响 Phase 门禁日期，调整本计划的时间线

典型场景举例：
- `chatStore.ts` 依赖 `runtimeStore` → Phase 2 时发现需要额外改造 → 评估后纳入 Phase 4 组件切换
- `executionStore.ts` 依赖旧 Store → 确认属于后续迭代
