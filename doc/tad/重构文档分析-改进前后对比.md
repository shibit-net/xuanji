### 4.2 module-coordination-plan 新增的价值

**1. IPC 协议变更明确化**（之前文档最大的缺口）

之前文档只说了「统一为一条通道」，但未说明：
- main process 如何处理新通道的请求
- preload.ts 怎么改
- 事件下行（Child→Renderer）的事件名如何调整

module-coordination-plan 用完整的代码对照解决了这个问题。

**2. 组件影响清单完整化**

清单了 18+ 个组件的改造方案，其中多个核心组件（ExecutionFlow、MessageBubble、WorkspaceMonitor、InputArea）给出了完整的代码对照。这大幅降低了实际改造中「不知道从哪改起」的风险。

**3. 实操层面的分批时序**

之前文档的 Phase 划分（新建→并行→清理）太粗，module-coordination-plan 将其细化到周级 + 组件级切换顺序，并配套了 feature flag 控制表，使得改造可以逐组件切换、逐组件回滚。

**4. 端到端验证清单**

7 个核心用户场景的逐步骤验证点 + IPC 事件验证 + Store 一致性验证，填补了之前文档缺失的质量保障环节。

### 4.3 module-coordination-plan 仍存在的小缺口

| 缺口 | 说明 | 建议 |
|------|------|------|
| `agent:thinking-start` 弃用策略不够明确 | 第 2.3 节说「弃用」「合并」，但没有说明后端怎么停发、前端怎么兼容过渡 | 补充 Phase 2 期间后端两头发送（新旧事件并行），Phase 3 删除旧事件 |
| **Store 间交叉通知的实现风险** | 第 3.2 节用内部回调实现 cross-store 通知，但回调在 Store 内部持有对其它 Store 的引用，可能导致循环依赖 | 补充 `CrossStoreEventBus` 或 Zustand middleware 的替代方案对比 |
| **Feature Flag 切换的测试策略** | 第 9.3 节列出了 5 个 flag，但未说明如何在 CI 中验证「flag 开/关」两种状态 | 补充 CI 矩阵：flag 全关 / flag 全开 / 混合模式 |

## 附录：代码级交叉验证发现

### 1. `ChatSession._pendingQueue` 同时被两个地方写入

**问题**：`_pendingQueue` 是 `ChatSession` 的消息队列，但它被**直接传引用**注入到 `AgentLoop`（line 66: `agentLoop.setPendingQueue(this._pendingQueue)`），同时 `ChatSession` 自身也在 `drainPendingQueue()`/`interrupt()`/`appendMessage()` 中操作同一个数组。这意味着消息队列被两个对象以引用共享的方式直接操作，破坏了单一职责。

**文档中是否覆盖**：backend-migration-plan 步骤 5.2 提到「消除 checkShouldStop 中的复杂判断」，但没有提及 `setPendingQueue` 引用注入带来的耦合。

**对迁移方案的影响**：新的 `SessionStateMachine` 必须同时接管 `AgentLoop` 对 `_pendingQueue` 的直接读取，消除引用注入。

### 2. `EventForwarder` 的 register 方法签名未明确

**问题**：backend-migration-plan.md 步骤 3 的 `EventForwarder` 代码中，映射表包含 `XuanjiEvent.CONVERSATION_STATE_CHANGED`、`XuanjiEvent.ASYNC_TASK_STARTED` 等事件。但实际 `EventBus.events.ts` 中的命名可能与文档不一致，需要确认事件名映射。

**建议**：补充 `EventForwarder.register()` 的完整映射表，标注每个 `XuanjiEvent.*` 的 source 位置。

### 3. `AgentLoop.setPendingQueue` 方法需要迁移

**问题**：当前 `AgentLoop` 通过 `setPendingQueue` 接收 `_pendingQueue` 的引用，在迭代边界检查（`checkShouldStop`）中直接读取长度。新的 `SessionStateMachine` 要替代这个机制，但文档未说明：
- 中间态：新旧两个系统共存时，`AgentLoop` 应该读谁的队列？
- 切换时机：`setPendingQueue` 在 `SessionStateMachine` 启用后何时删除？

### 4. `drainPendingQueue` 的 `_drainRunning` guard 未被分析

**问题**：`ChatSession` 中有 `_drainRunning` 布尔值（line 37）防止 `drainPendingQueue` 重入。文档分析了一、二、三层 reentrancy guard，但未提及这个第四层 guard。虽然这个 guard 影响较小，但重构时应注意是否需要保留。

### 5. `hook` 系统与 `EventBus` 的边界未明确

**问题**：实际代码中 `src/hooks/HookRegistry.ts`（304 行）和 `src/core/events/EventBus.ts`（231 行）是两个独立的事件系统：
- `EventBus` — 核心架构事件（AGENT_STARTED, TEXT_DELTA, ASYNC_TASK_COMPLETED 等）
- `HookRegistry` — 用户可配置的 Hook 事件（通过 `HookEventEmitter` 实现，支持命令 handler、prompt handler、agent handler）

`agent-bridge.ts` 中这两者都使用了（registerHookEventBridge 负责 EventBus→IPC，hook 注册负责 HookRegistry→agent 生命周期扩展）。文档未讨论这两套事件系统的关系，重构后的 `EventForwarder` 是否应接管所有事件转发也存疑。

### 6. `useAgentManager` hook 的 IPC 调用未被覆盖

**问题**：`desktop/renderer/hooks/useAgentManager.ts` 直接调用了 `window.electron.agentList()`、`window.electron.agentCreate()` 等 CRUD IPC 方法。这些 IPC 通道与 `agent:send-message`/`agent:interrupt` 等核心通道无关，但 module-coordination-plan 中未在做 preload.ts 改造时确认这些非核心通道是否受影响。

**确认结果**：这些 IPC 方法与 `agent:user-action` 改造无关，**不受影响**。但建议在附录中明确标注。

**偏差纠正**：对比文档中「**preload.ts 改造**」需要 +1 项待确认项。确认后这些方法不变，文档的 preload 改造代码只需新增而不需修改。
