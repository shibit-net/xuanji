### ⚠️ "错误"案例分析（2个）

#### 案例1: "分析 src/utils/helper.ts 的代码质量"

**我的预期**: direct（单文件分析）
**LLM 判断**: decompose (medium, 4步)

**LLM 的理由**:
> 任务需要读取文件、分析代码质量（包括可读性、性能、最佳实践等）、生成评估报告。涉及代码审查专业知识，但步骤数在中等范围内，子任务需按顺序执行。

**反思**:
- ✅ LLM 识别出代码质量分析涉及**多个维度**（可读性、性能、最佳实践）
- ✅ LLM 识别出需要**专业知识**（代码审查）
- ✅ 4步虽然<5，但触发了 `requiresSpecialist` 条件
- **结论**: 这不是错误，LLM 的判断更准确

#### 案例2: "实现用户注册功能"

**我的预期**: direct（单一功能）
**LLM 判断**: decompose (medium, 6步)

**LLM 的理由**:
> 用户注册功能涉及多个顺序步骤：需求分析、数据库设计、后端API实现、前端表单、验证逻辑和测试。虽然步骤数在中等范围，但不需要专业Agent，通用AI助手可完成。子任务间存在依赖关系，难以并行执行。

**LLM 的分解**:
1. 需求分析
2. 数据库设计
3. 后端API实现
4. 前端表单
5. 验证逻辑
6. 测试

**反思**:
- ✅ LLM 正确识别出"用户注册"包含**前后端多层**
- ✅ 6步≥阈值5，触发分解模式
- ✅ "单一功能"不等于"简单任务"
- **结论**: 这不是错误，LLM 比我的预期更细致

## 💡 关键发现

### 1. LLM 判断超越规则

**LLM 不只看步骤数，还综合判断**:
- ✅ 是否涉及专业领域（代码审查、安全分析、性能优化）
- ✅ 是否可分解为独立子任务
- ✅ 任务复杂度是否超出通用能力
- ✅ 子任务间的依赖关系

**示例对比**:
| 任务 | 步骤数 | LLM判断 | 理由 |
|------|--------|---------|------|
| 修复bug | 4步 | direct | 通用能力可完成 |
| 分析代码质量 | 4步 | decompose | 需要专业知识 |
| 重构单函数 | 4步 | direct | 标准重构工作 |
| 实现注册功能 | 6步 | decompose | 涉及前后端多层 |

### 2. 边界case智能处理

**中等任务（4-6步）的判断模式**:
```
步骤数 + 专业性 + 领域数 → 决策

示例:
4步 + 通用能力 + 单领域 → direct
4步 + 专业知识 + 多领域 → decompose
6步 + 任何情况 → decompose
```

### 3. 当前阈值设计合理

```typescript
const needsDecompose =
  complexity.complexity === 'complex' ||     // 复杂度为 complex
  complexity.requiresSpecialist ||           // 需要专业能力 ⭐
  complexity.estimatedSteps >= 5;            // 步骤数 ≥5
```

这个三条件设计很好：
- ✅ 不只依赖步骤数
- ✅ 考虑专业性（两个"错误"都由此触发）
- ✅ 给LLM灵活判断的空间

## 📝 实施记录

### 修复的核心问题

**问题**: ComplexityAnalyzer 调用失败，全部降级到 simple

**根本原因**: Provider 配置传递缺失
```typescript
// ❌ 之前：缺少 apiKey
const stream = this.provider.stream(messages, tools, {
  model: this.model,
  maxTokens: 500,
});

// ✅ 现在：传入完整配置
const stream = this.provider.stream(messages, tools, {
  ...this.providerConfig,  // apiKey, baseURL, timeout
  model: this.model,
  maxTokens: 500,
});
```

### 修改的文件

1. **ComplexityAnalyzer.ts**: 新增 `providerConfig` 参数
2. **TaskRouter.ts**: 新增 `providerConfig` 参数并传递
3. **ChatSession.ts**: 在创建 TaskRouter 时传入配置
4. **test/benchmarks/**: 新增诊断和基准测试脚本
