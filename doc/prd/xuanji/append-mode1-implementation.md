# 补充输入方式1实现 - 立即中断，清除重来

## 执行时间
2026-03-05 13:15

## 用户需求

用户期望的补充输入行为：

**场景**：
1. 用户：`总结一下项目结构`
2. xuanji 流式输出：`项目结构包括 src/、doc/、test/...`
3. 用户补充：`使用英文总结`
4. **期望行为**：立即中断当前输出，清除已输出内容，基于原始问题 + 补充重新生成完整答案

**效果**：
```
用户：总结一下项目结构
助手：项目结构包括 src/、doc/... ← 被中断，清除
用户：使用英文总结               ← 补充输入
助手：Project structure includes src/, doc/, test/... ← 重新生成（完整的英文答案）
```

**关键特点**：
- ✅ 已输出的中文内容被**清除**（不保留）
- ✅ 只有一段输出（英文），无冗余
- ✅ 基于完整上下文（原始问题 + 补充）重新生成
- ✅ 符合"改变主意"的直觉

## 核心改动

### 改动1：清除流式文本，不归档（L1857-1862）

**修改前**（方式2：保留已输出内容）：
```typescript
// interrupt 前归档流式文本
archiveStreamText();  // 保留已输出的中文内容
```

**修改后**（方式1：清除重来）：
```typescript
// ★ 方式1：立即中断，清除重来 ★
// 清除已输出的流式文本（不保留，重新生成）
streamTextRef.current = '';
setStreamText('');
```

**效果**：
- 已输出的内容不会被保留到历史消息
- UI 上会立即清空流式输出区域
- 新的响应会完全替换旧内容

---

### 改动2：补充输入不添加特殊标记（L1872）

**修改前**：
```typescript
content: `💬 ${pending.content}`,  // 带表情符号标记
```

**修改后**：
```typescript
content: pending.content,  // 无需特殊标记（因为清除重来）
```

**理由**：
- 方式1 清除重来，不会有"割裂"感
- 补充输入就是普通的 user 消息
- 无需特殊标记来区分

---

### 改动3：补充输入时不归档（L1792-1800）

**修改前**：
```typescript
if (streamTextUpdaterRef.current) {
  streamTextUpdaterRef.current.flush();
}
archiveStreamText();  // 归档到历史
```

**修改后**：
```typescript
if (streamTextUpdaterRef.current) {
  streamTextUpdaterRef.current.flush();
}
// ★ 方式1：不归档，保持在 streamText 中（后续会清除） ★
// 这样用户能看到正在输出的内容，直到 interrupt 时清除重来
```

**效果**：
- 用户输入补充时，已输出的内容仍然可见
- 100ms debounce 期间，用户能看到"正在输出"
- interrupt 触发后，内容被清除，重新生成

## 完整流程

### 用户体验流程

```
1. 用户输入："总结一下项目结构"
   ↓
2. xuanji 开始流式输出（中文）
   "项目结构包括以下几个部分：
    1. src/ - 源代码目录
    2. doc/ - 文档..."
   ↓
3. 用户补充："使用英文总结"
   ↓
4. UI 立即显示：
   - 流式输出区域：仍显示已输出的中文内容（100ms 内）
   - 动态提示："✓ 已收到 1 条补充：使用英文总结"
   ↓
5. 100ms 后（debounce 触发）：
   - 清除流式输出区域（中文内容消失）
   - 补充输入添加到历史："使用英文总结"
   - 显示处理中提示："⏳ 正在处理补充输入... 使用英文总结"
   ↓
6. interrupt 触发 → LLM 重新生成
   - 基于完整消息历史：
     ["总结一下项目结构", "使用英文总结"]
   ↓
7. 新的流式输出开始（英文）
   - 处理中提示消失
   - 输出："Project structure includes the following components:
            1. src/ - Source code directory
            2. doc/ - Documentation..."
```

### 技术流程

```typescript
// 1. 用户输入补充
handleSubmit("使用英文总结")
  ↓
// 2. flush 流式文本更新器
streamTextUpdater.flush()
  ↓
// 3. 不归档（保持在 streamText 中）
// archiveStreamText();  ← 删除
  ↓
// 4. 添加到 pending 队列
setPendingUserInputs([{ content: "使用英文总结", timestamp }])
  ↓
// 5. 设置 debounce 定时器（100ms）
setTimeout(() => {
  // 6. 清除流式文本
  streamTextRef.current = '';
  setStreamText('');

  // 7. 添加补充输入到历史
  setMessages([...prev, { role: 'user', content: "使用英文总结" }]);

  // 8. 设置处理中状态
  setProcessingAppend("使用英文总结");

  // 9. 调用 interrupt
  agentLoop.interrupt("使用英文总结");
}, 100);
  ↓
// 10. LLM 重新生成
// 消息历史：["总结一下项目结构", "使用英文总结"]
// 生成新的完整答案（英文）
  ↓
// 11. onThinking 触发
// 清除 processingAppend 状态
setProcessingAppend(null);
  ↓
// 12. 新的流式输出开始
// 输出英文总结
```

## 与方式2的对比

| 特性 | 方式1（当前实现） | 方式2（之前实现） |
|------|------------------|------------------|
| **已输出内容** | ❌ 清除，不保留 | ✅ 归档，保留在历史 |
| **输出段数** | 1 段（重新生成） | 2 段（原内容 + 新内容） |
| **冗余内容** | ❌ 无冗余 | ⚠️ 可能有冗余（中英文都有） |
| **适用场景** | "改变方向"（如切换语言） | "追加细节"（如添加示例） |
| **符合直觉** | ✅ 更符合"改变主意" | ⚠️ 更像"追加内容" |
| **Token 消耗** | ✅ 更低（只生成一次） | ⚠️ 更高（生成两次） |
| **实现复杂度** | ✅ 更简单（删除归档） | ⚠️ 稍复杂（需要连接语） |

## 边界情况处理

### 情况1：补充输入后立即按 Ctrl+C

**流程**：
```
用户补充 → debounce 100ms → （用户按 Ctrl+C）→ interrupt 未触发
```

**处理**：
- handleInterrupt 会检查 pending 队列
- 将 pending 输入添加到历史
- 显示中断提示

**结果**：补充输入不会丢失，会在中断时添加到历史。

---

### 情况2：快速连续补充（< 100ms）

**流程**：
```
用户补充1 → 50ms → 用户补充2 → debounce 重置 → 100ms → interrupt
```

**处理**：
- 第一次补充设置 debounce 定时器
- 第二次补充取消定时器，重新设置
- 最终只触发一次 interrupt（基于最后的补充）

**结果**：符合预期，避免频繁 API 调用。

---

### 情况3：工具执行期间补充

**流程**：
```
status='tool' → 用户补充 → appendMessage（温和追加）
```

**处理**：
- 不调用 interrupt（因为 status !== 'thinking'）
- 使用 appendMessage 温和追加
- 工具继续执行，不中断

**结果**：工具执行不受影响，补充输入在工具完成后处理。

---

### 情况4：补充输入后 API 错误

**流程**：
```
补充输入 → interrupt → API 请求失败 → onError
```

**处理**：
- onError 清除 processingAppend 状态
- 显示错误消息
- 补充输入已在历史中，不会丢失

**结果**：用户可以重试或继续对话。

## 已知限制

### 限制1：已输出内容不保留

**影响**：
- 如果已输出大量有用内容（如长篇总结），补充输入会导致全部重来
- 用户可能觉得"浪费"

**缓解**：
- 方式1 更适合"改变方向"的补充（如切换语言、格式）
- 如果用户需要"追加内容"，可以等输出完成后再输入

**未来改进**：
- 可考虑支持快捷键切换方式1/方式2
- 或根据补充内容自动判断（如包含"英文"→方式1，"添加"→方式2）

---

### 限制2：100ms debounce 内容仍可见

**影响**：
- 用户输入补充后，已输出的内容仍显示 100ms
- 可能产生短暂的视觉残留

**缓解**：
- 100ms 非常短，几乎无感知
- 显示"✓ 已收到补充"提示，告知用户已接收

**未来改进**：
- 可考虑立即清除（无 debounce），但会增加 API 调用频率
