# 补充输入功能 - 完整总结

## 执行时间
2026-03-05 13:20

## 功能概述

xuanji 支持在 Agent 执行期间（流式输出或工具执行时）输入补充内容，实现**立即中断，清除重来**的交互模式。

---

## 用户场景

### 场景示例

```
用户：总结一下 xuanji 项目的核心功能
助手：xuanji 是一个基于 Claude API 的 AI 助手...
      核心功能包括：
      1. 会话管理 - 支持保存和恢复对话
      2. 记忆系统 - ...

用户：使用英文总结  ← 补充输入

助手：（清除已输出的中文内容）
      Xuanji is a Claude API-based AI assistant...
      Core features include:
      1. Session Management - ...
      2. Memory System - ...
```

### 核心特点

- ✅ **立即响应**：补充输入后 100ms 内触发中断
- ✅ **清除重来**：已输出内容被清除，基于完整上下文重新生成
- ✅ **无冗余**：只保留最终输出，无中间内容残留
- ✅ **明确反馈**：显示"正在处理补充输入..."提示

---

## 实现方式

### 方式1：立即中断，清除重来（当前实现）

**特点**：
- 补充输入后，立即停止流式输出
- 清除已输出内容（不保留）
- 基于原始问题 + 补充重新生成完整答案
- 适合"改变方向"的补充（如切换语言、格式）

**优势**：
- ✅ 无冗余内容
- ✅ Token 消耗更低
- ✅ 符合"改变主意"的直觉
- ✅ 更像 Claude Code 的行为

**劣势**：
- ⚠️ 已输出的内容丢失（可能觉得"浪费"）
- ⚠️ 不适合"追加细节"的场景

---

### ~~方式2：继续输出 + 追加（已废弃）~~

**特点**：
- 保留已输出内容，归档到历史
- 补充输入作为新的 user 消息
- 继续生成新内容，追加在同一轮对话中

**优势**：
- ✅ 保留已生成的内容（不浪费）
- ✅ 适合"追加细节"的场景

**劣势**：
- ⚠️ 可能有冗余（如中英文都保留）
- ⚠️ Token 消耗更高
- ⚠️ 视觉上割裂（像两轮对话）

**状态**：已废弃，用户选择方式1。

## 技术实现

### 核心逻辑

```typescript
// 用户输入补充内容
if (status === 'thinking') {
  // 1. flush 流式文本更新器
  streamTextUpdater.flush();

  // 2. 不归档（方式1：清除重来）
  // archiveStreamText();  ← 删除

  // 3. 添加到 pending 队列
  setPendingUserInputs([{ content: input, timestamp }]);

  // 4. 100ms debounce 后触发 interrupt
  setTimeout(() => {
    // 清除已输出的流式文本
    streamTextRef.current = '';
    setStreamText('');

    // 添加补充输入到历史
    setMessages([...prev, { role: 'user', content: input }]);

    // 设置"处理中"状态
    setProcessingAppend(input);

    // 调用 interrupt
    agentLoop.interrupt(input);
  }, 100);
}
```

### 状态管理

| 状态 | 含义 | 设置时机 | 清除时机 |
|------|------|---------|---------|
| `pendingUserInputs` | 补充输入队列 | 用户输入时 | interrupt 触发后 |
| `processingAppend` | 处理中的补充输入 | interrupt 触发时 | onThinking / onError / onEnd |
| `streamText` | 当前流式输出 | 流式输出时 | interrupt 触发时 / onEnd |

### 关键时机

```
用户输入补充
  ↓
flush 流式文本
  ↓
添加到 pending 队列
  ↓
显示"✓ 已收到补充"提示
  ↓
100ms debounce
  ↓
清除流式文本
  ↓
添加补充输入到历史
  ↓
设置 processingAppend 状态
  ↓
显示"⏳ 处理中"提示
  ↓
调用 interrupt
  ↓
LLM 重新生成
  ↓
onThinking 触发
  ↓
清除 processingAppend
  ↓
新的流式输出开始
```

## 边界情况

### 工具执行期间补充

- **行为**：使用 `appendMessage`（温和追加），不中断工具
- **原因**：工具执行不应被中断
- **效果**：工具完成后处理补充输入

### 快速连续补充

- **行为**：100ms 内的补充会重置 debounce 定时器
- **效果**：只触发一次 interrupt（基于最后的补充）
- **目的**：避免频繁 API 调用

### 补充后按 Ctrl+C

- **行为**：`handleInterrupt` 会将 pending 队列添加到历史
- **效果**：补充输入不会丢失
- **结果**：显示中断提示，补充输入在历史中

### API 错误

- **行为**：`onError` 清除 `processingAppend` 状态
- **效果**：错误提示显示，补充输入已在历史中
- **结果**：用户可以重试或继续对话

## 性能优化

### Debounce 机制

- **延迟**：100ms
- **目的**：避免快速连续输入时触发多次 API 调用
- **效果**：Rate limit 风险降低

### 队列合并

- **窗口**：3 秒
- **规则**：3 秒内的连续补充合并为一条
- **效果**：减少消息数量，简化历史

## 已修复的问题

### 问题1：流式输出上下文割裂（已修复）

**现象**：
- 补充输入后，已输出内容和新内容视觉上割裂
- 用户不清楚它们的关系

**修复**：
- 方式1：清除已输出内容，重新生成
- 视觉上连贯，无割裂感

---

### 问题2：处理反馈不明确（已修复）

**现象**：
- interrupt 后到新响应前，无任何反馈
- 用户不知道补充输入是否被处理

**修复**：
- 新增 `processingAppend` 状态
- 显示"⏳ 正在处理补充输入..."提示
- 提示持续到新响应开始

---

### 问题3：第二次补充无响应（已修复）

**现象**：
- 第一次补充正常，第二次补充无响应
- 日志：`appendMessage() called but not running, ignoring`

**原因**：
- `agentLoop.running` 和 UI `status` 之间存在状态不一致窗口

**修复**：
- 补充输入前检查 `agentLoop.getState().status`
- onEnd 时自动处理 pending 队列
