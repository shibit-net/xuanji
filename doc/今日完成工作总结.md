## 2. 统一核心规则和身份记忆系统 ✅

### 设计决策
将 CoreRuleStore 和 IdentityManager 统一到 memory.db，使用统一的 PermanentConstraintManager 接口。

### 架构设计

```
PermanentConstraintManager (统一接口)
    ↓
MemoryStore (memory.db)
    ↓
所有永久约束存储为：
- constraint: 'must'
- volatility: 'permanent'
- scope: 'core_rule' | 'profile'
```

### 实现内容

#### 新增组件

1. **PermanentConstraintManager** - 统一的永久约束管理器
   - 管理所有必须遵守的永久约束
   - 支持身份记忆（用户称呼、助手名字）
   - 支持核心规则（行为约束、隐私规则等）
   - 统一存储在 memory.db

2. **ConstraintClassifier** - 智能分类器
   - 使用 LLM 判断是永久约束还是普通记忆
   - 支持快速关键词匹配（90%+ 置信度）
   - 支持语义判断（70%+ 置信度）

3. **SmartMemoryStorage** - 智能存储器
   - 自动分类并存储用户输入
   - 根据置信度决定存储类型
   - 支持用户确认（置信度 < 70%）

#### 兼容性保证

**CoreRuleStore** - 改为适配器模式
- 保持所有现有接口不变
- 内部委托给 PermanentConstraintManager
- 所有数据存储在 memory.db（不再使用 core-rules.json）

**IdentityManager** - 保持独立
- 继续提供结构化 API
- 内部使用 MemoryStore 存储
- 与 PermanentConstraintManager 共享数据

### 区分机制

#### 永久约束 vs 普通记忆

| 特性 | 永久约束 | 普通记忆 |
|------|---------|---------|
| constraint | must | may/should |
| volatility | permanent | stable/normal/transient |
| scope | core_rule/profile | knowledge/episode |
| 衰减 | 永不衰减 | 会随时间衰减 |
| 注入 | 始终注入 | 根据相关性检索 |
| 优先级 | 最高 | 动态权重 |

#### 判断流程

```
用户输入
    ↓
1. 用户明确指令？(/rule, /identity)
    ↓ 是 → 存储为约束
    ↓ 否
2. 快速关键词匹配（90%+）
    ↓ 匹配 → 存储为约束/记忆
    ↓ 否
3. LLM 语义判断（70%+）
    ↓ 置信度高 → 存储为约束/记忆
    ↓ 置信度低
4. 询问用户确认
    ↓
存储到 memory.db
```
