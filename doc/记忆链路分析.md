# 记忆/规则/身份 链路分析

## 概览

璇玑的记忆系统分为三个核心概念：**记忆（Memory）**、**规则（Rule/Constraint）**、**身份（Identity）**。它们在存储层统一，但在语义层和使用场景上有明确区分。

---

## 一、存储层架构

### 1.1 统一存储：memory.db (SQLite)

所有记忆、规则、身份信息都存储在同一个 SQLite 数据库中，通过以下字段区分：

| 字段 | 作用 | 取值 |
|------|------|------|
| `type` | 记忆类型 | `user_fact`, `user_preference`, `relationship`, `important_date`, `decision`, `tool_pattern`, `error_resolution` |
| `scope` | 作用域 | `core_rule`（核心规则）, `profile`（用户画像）, `knowledge`（经验知识）, `episode`（近期上下文） |
| `volatility` | 持久性 | `permanent`（永久）, `stable`（稳定）, `normal`（普通）, `transient`（临时） |
| `constraint` | 约束级别 | `must`（必须遵守）, `should`（应该遵守）, `may`（可选参考） |

### 1.2 核心组件

```
MemoryManager (协调器)
├── MemoryStore (SQLite 存储)
├── PermanentConstraintManager (永久约束管理)
│   └── 管理所有 constraint='must' + volatility='permanent' 的记忆
├── CoreRuleStore (兼容层，委托给 PermanentConstraintManager)
├── IdentityManager (身份管理)
│   └── 管理 type='user_fact' 且包含身份关键词的记忆
├── ConstraintClassifier (智能分类器)
│   └── 使用 LLM 判断用户输入是约束还是普通记忆
└── SmartMemoryStorage (智能存储)
    └── 根据分类结果自动路由到对应管理器
```

## 二、写入链路（存储）

### 2.1 实时触发链路（新增，2026-04-18）

**触发条件**：用户输入包含记忆关键词

```
用户输入："记住，以后回复要简洁"
    ↓
SessionOrchestrator.execute()
    ↓ 关键词检测
SessionOrchestrator.detectMemoryIntent()
    ↓ 检测到约束类关键词：'记住'
返回提示："IMPORTANT: The user is stating a constraint..."
    ↓
PromptOrchestrator.buildAndApply(userMessage, memoryHint)
    ↓ 注入记忆提示到 system prompt
AgentLoop.run()
    ↓ LLM 看到提示，主动调用工具
MemoryStoreTool.execute({ type: 'user_preference', content: '回复要简洁' })
    ↓
MemoryStore.saveEntry()
    ↓ 存入 memory.db
    constraint: 'must'
    volatility: 'permanent'
    scope: 'core_rule'
```

**关键词列表**：
- 约束类：记住、记得、以后、规则、约束、不要、必须、remember、always、never、rule、constraint
- 偏好类：喜欢、偏好、习惯、prefer、like、habit
- 身份类：叫我、称呼、名字、叫、call me、name

**特点**：
- ✅ 实时生效（当轮就能存储）
- ✅ 零成本过滤（90%+ 普通对话直接跳过）
- ✅ 利用现有工具（MemoryStoreTool）

### 2.2 会话结束链路（原有机制）

**触发条件**：会话结束

```
ChatSession.run() 完成
    ↓
TurnLifecycleManager.afterTurn()
    ↓ 判断是否需要归档
MemoryFlushAgent (SubAgent)
    ↓ 分析整个对话历史
提取记忆 → MemoryStore.saveEntry()
    ↓ 批量存入 memory.db
根据内容自动标记：
    - constraint: 'must' / 'should' / 'may'
    - volatility: 'permanent' / 'stable' / 'normal' / 'transient'
    - scope: 'core_rule' / 'profile' / 'knowledge' / 'episode'
```

**特点**：
- ⏰ 延迟生效（会话结束后才提取）
- 🧠 智能分析（LLM 深度理解对话）
- 📦 批量处理（一次性提取多条记忆）

### 2.3 显式命令链路

**触发条件**：用户使用斜杠命令

```
用户输入："/rule add 回复要简洁"
    ↓
SkillRouter.tryRouteToSkill()
    ↓ 路由到 RuleSkill
RuleSkill.execute()
    ↓
CoreRuleStore.add({ rule: '回复要简洁', category: 'response_style' })
    ↓ 委托给 PermanentConstraintManager
PermanentConstraintManager.add()
    ↓
MemoryStore.saveEntry()
    ↓ 存入 memory.db
    constraint: 'must'
    volatility: 'permanent'
    scope: 'core_rule'
```

**特点**：
- ⚡ 立即生效
- 🎯 精确控制（用户明确指定类型）
- 🔧 高级用户专用

## 三、读取链路（使用）

### 3.1 每次回复前注入

**触发时机**：每次生成回复前

```
SessionOrchestrator.runSingleAgent()
    ↓
PromptOrchestrator.buildAndApply()
    ↓ 构建 system prompt
LayeredPromptBuilder.build()
    ↓ 生成基础 prompt
PromptOrchestrator 注入记忆：
    ↓
    ├─ MemoryManager.formatDecisionContext(userMessage)
    │   ↓ 检索相关记忆
    │   MemoryRetriever.retrieve()
    │       ↓ 向量检索 + 关键词匹配
    │       返回：核心规则 + 用户画像 + 相关经验 + 待处理事项
    │
    └─ IdentityManager.getIdentity()
        ↓ 查询身份记忆
        返回：assistantName, userTitle, persona, tone
        ↓
        IdentityManager.formatForSystemPrompt()
        ↓ 格式化为 prompt 文本
        "# Identity\n- Assistant name: 贾维斯\n- User title: Boss"
    ↓
最终 system prompt = 基础 prompt + DecisionContext + Identity
    ↓
AgentLoop.run() 使用最终 prompt 调用 LLM
```

### 3.2 记忆检索策略

**DecisionContext 包含四层**：

1. **核心规则层**（Core Rules）
   - 查询条件：`constraint='must'` + `volatility='permanent'`
   - 始终注入，无条件遵守
   - 示例：`回复要简洁`、`使用中文回复`

2. **用户画像层**（Profile）
   - 查询条件：`scope='profile'` + `volatility='stable'`
   - 用户的长期偏好和习惯
   - 示例：`喜欢吃辣`、`早上工作效率高`

3. **经验知识层**（Knowledge）
   - 查询条件：`scope='knowledge'` + 向量相似度
   - 历史决策、工具使用模式、错误解决方案
   - 示例：`使用 TypeScript 开发新项目`

4. **近期上下文层**（Episode）
   - 查询条件：`scope='episode'` + 时间近期
   - 最近的对话上下文
   - 示例：`正在重构 memory 模块`

**Identity 包含**：
- `assistantName`：助手名字（如：贾维斯）
- `userTitle`：用户称呼（如：Boss）
- `persona`：人格特质（如：专业、幽默）
- `tone`：语气风格（如：正式、轻松）

## 四、三者的区别

| 维度 | 记忆（Memory） | 规则（Rule/Constraint） | 身份（Identity） |
|------|---------------|------------------------|-----------------|
| **语义** | 事实性信息 | 行为约束 | 角色定位 |
| **示例** | "用户喜欢吃辣" | "回复要简洁" | "助手叫贾维斯，用户是Boss" |
| **存储标记** | `constraint='may'` | `constraint='must'` | `type='user_fact'` + 身份关键词 |
| **持久性** | `normal` / `transient` | `permanent` | `permanent` |
| **作用域** | `knowledge` / `episode` | `core_rule` | `profile` |
| **注入方式** | DecisionContext（按需检索） | DecisionContext（始终注入） | Identity 专属块（始终注入） |
| **管理器** | MemoryStore | PermanentConstraintManager | IdentityManager |
| **兼容层** | - | CoreRuleStore | - |

## 五、关键设计决策

### 5.1 为什么统一存储？

**优势**：
- 避免数据孤岛（规则和记忆可以互相引用）
- 统一的向量检索（规则也可以被语义检索）
- 简化维护（一个数据库，一套备份/迁移逻辑）

**如何区分**：
- 通过 `constraint`、`volatility`、`scope` 字段标记
- 通过专门的管理器（PermanentConstraintManager、IdentityManager）封装查询逻辑

### 5.2 为什么保留 CoreRuleStore？

**向后兼容**：
- 旧代码中大量使用 `CoreRuleStore` API
- 通过适配器模式，底层委托给 `PermanentConstraintManager`
- 用户无感知迁移

### 5.3 为什么需要 ConstraintClassifier？

**智能分类**：
- 用户说"记住，以后要用中文"，系统需要判断这是规则还是普通记忆
- 使用 LLM 语义分析，准确率 > 90%
- 快速关键词匹配 + LLM 深度分析的混合策略

### 5.4 为什么需要实时触发链路？

**用户体验**：
- 旧机制：用户说"记住 X"，要等会话结束才生效 ❌
- 新机制：关键词检测 → 立即提示 LLM → 当轮存储 ✅
- 成本可控：90% 的对话通过关键词直接过滤，零 LLM 调用

## 六、完整示例

### 示例 1：用户设置规则

```
用户: 记住，以后回复要简洁

[实时触发链路]
1. SessionOrchestrator 检测到关键词 '记住'
2. 注入提示到 system prompt
3. LLM 调用 memory_store 工具
4. 存入 memory.db:
   {
     type: 'user_preference',
     content: '回复要简洁',
     constraint: 'must',
     volatility: 'permanent',
     scope: 'core_rule'
   }

[下次对话]
5. PromptOrchestrator 读取 DecisionContext
6. 注入到 system prompt: "Core Rules: 回复要简洁"
7. LLM 生成简洁回复
```

### 示例 2：用户设置身份

```
用户: 叫我 Boss，你叫贾维斯

[实时触发链路]
1. SessionOrchestrator 检测到关键词 '叫我'、'你叫'
2. 注入提示到 system prompt
3. LLM 调用 memory_store 工具两次:
   - memory_store({ type: 'user_fact', content: '用户称呼是 Boss' })
   - memory_store({ type: 'user_fact', content: '助手名字是贾维斯' })
4. 存入 memory.db

[下次对话]
5. IdentityManager.getIdentity() 查询身份记忆
6. 解析出: assistantName='贾维斯', userTitle='Boss'
7. 注入到 system prompt:
   "# Identity
   - Assistant name: 贾维斯
   - User title: Boss"
8. LLM 回复: "好的 Boss，贾维斯随时为您服务"
```

### 示例 3：用户分享偏好

```
用户: 我喜欢吃辣

[实时触发链路]
1. SessionOrchestrator 检测到关键词 '喜欢'
2. 注入提示到 system prompt
3. LLM 调用 memory_store 工具
4. 存入 memory.db:
   {
     type: 'user_preference',
     content: '喜欢吃辣',
     constraint: 'may',
     volatility: 'stable',
     scope: 'profile'
   }

[未来对话：推荐餐厅]
用户: 推荐一家餐厅

5. MemoryRetriever 向量检索 "推荐 餐厅"
6. 匹配到 "喜欢吃辣"（相关性高）
7. 注入到 DecisionContext: "User Profile: 喜欢吃辣"
8. LLM 回复: "推荐川菜馆，符合您的口味"
```

## 七、未来优化方向

### 7.1 Memory Agent 架构（待实现）

**当前问题**：
- 实时触发依赖关键词，可能漏检测
- 会话结束触发延迟，用户体验不佳

**改进方案**：
```
用户输入
    ↓
主 Agent 判断：是否需要记忆？
    ↓ 如果需要
调用 Memory Agent (SubAgent)
    ↓
Memory Agent 分析：
    - 是约束还是记忆？
    - 约束级别？（must/should/may）
    - 持久性？（permanent/stable/normal）
    ↓
立即存储到 memory.db
    ↓
主 Agent 继续执行
```

**优势**：
- 主 Agent 主动判断，不依赖关键词
- Memory Agent 专业分析，准确率更高
- 实时存储，立即生效

### 7.2 记忆冲突检测

**场景**：
- 旧记忆："用户不吃辣"
- 新记忆："用户喜欢吃辣"

**改进**：
- 存储前检测冲突
- 提示用户确认
- 自动标记旧记忆为 `obsolete`

### 7.3 记忆重要性自动调整

**场景**：
- 用户多次提到某个偏好 → 提升 `constraint` 级别
- 长期未使用的记忆 → 降低权重或归档

**改进**：
- 记录记忆使用频率
- 定期重新评估重要性
- 自动调整 `constraint` 和 `volatility`

## 八、总结

璇玑的记忆系统通过**统一存储 + 语义区分**的架构，实现了记忆、规则、身份的协同管理：

1. **存储层统一**：所有数据存储在 memory.db，通过字段标记区分
2. **语义层分离**：通过专门的管理器（PermanentConstraintManager、IdentityManager）封装不同语义
3. **写入多链路**：实时触发（关键词）+ 会话结束（LLM 分析）+ 显式命令（斜杠命令）
4. **读取智能化**：向量检索 + 分层注入，确保相关记忆始终可用
5. **向后兼容**：通过适配器模式（CoreRuleStore）保持旧代码兼容

这套架构既保证了灵活性（统一存储），又保证了易用性（语义分离），为未来的 Memory Agent 架构奠定了基础。
