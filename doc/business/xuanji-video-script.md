## 脚本正文

### 场景 0：开场 Hook（0:00 - 0:25）

> 画面：黑屏渐入，代码滚动背景，Xuanji logo 居中

**旁白**：
你有没有想过，一个 AI 助手，真的能像一支团队一样工作？

我是说，不是那种「调一个 API，等它慢慢输出」——

而是：你一句话，它自己分解任务，自己组建小队，
有人写代码，有人审查，有人辩论架构，
做完之后，还能自动回来告诉你结果。

这就是 Xuanji。璇玑，开源的个人 AI 代理。

今天我不聊那些高大上的架构图，
我用 3 分钟，讲明白 Xuanji 的核心设计理念——
以及它和 Hermes Agent、OpenCLAW 之间，
最关键的几个区别。

---

### 场景 1：我们要解决什么（0:25 - 0:55）

> 画面：三个并列截图——Cursor IDE / Claude Code CLI / ChatGPT Desktop

**旁白**：
现在市面上的 AI 工具有三种：

Cursor / Copilot，是 IDE 插件——帮你补全代码。
Claude Code，是终端助手——给你一个命令行 ReAct 循环。
ChatGPT Desktop，是聊天机器人——你一句它一句。

它们有一个共同的问题：
所有工作都是「一个人」做。

当你要重构一个模块、审查代码、做架构决策的时候，
你只能等它一步一步推理。

**Xuanji 的核心理念**：
> 一个 AI 助手不是「一个大脑」，而应该是一支「团队」。

---

### 场景 2：Xuanji 是怎么做的（0:55 - 1:50）

> 画面：五个 agent_team 策略的动态流程图动画（建议用代码可交互的 3D 或 motion graphic）

**旁白**：
Xuanji 最核心的设计，叫做 agent_team。

它不是一个工具，是一个**团队协作框架**。
五种策略：

并行（parallel）——三个 agent 同时写不同模块，互不等待。
顺序（sequential）——让一个 agent 的输出成为另一个的输入。
流水线（pipeline）——像工厂流水线，A 做完交给 B。
层级（hierarchical）——一个 leader 动态派任务给 workers。
以及最特别的：辩论（debate）。

> 画面：辩论模式的放大动画——正方、反方、裁判，三个气泡你来我往

比如你做架构决策，
一个 agent 说「用微服务」，
另一个说「不行，单体就够了」，
第三个裁判综合双方论点给出结论。
这就是 Xuanji 的辩论模式。

> 画面切入对比：Hermes 的 delegate_task 只有一个框

相比之下，Hermes Agent 只有一个 `delegate_task`——
扔一个 goal 出去，等结果回来。简单，但粗糙。
OpenCLAW 的 .agents 目录也是类似，没有策略、没有辩论、没有角色配置。

**Xuanji，是第一个把这个复杂度做进系统里的。**

---

### 场景 3：记忆系统——不是 KV 存储，是推理链路（1:50 - 2:35）

> 画面：一张拓扑图动画——节点「用户」→「不喜欢」→「川菜」→「辣」→「湘菜」

**旁白**：
你可能看过不少 AI 助手说「我有记忆」。
大多时候就是存个 key-value：「用户不喜欢川菜」。

Xuanji 不一样。
它存的是**拓扑图**。

你说「我不吃川菜，因为太辣了」——
它不是存一句话。
它分解成：

- 实体：川菜，辣
- 关系：你 → 不喜欢 → 川菜；你 → 不喜欢 → 辣；川菜 → 具有属性 → 辣

然后有一天你说「朋友请我吃湘菜」，
它能自己推理出来——
湘菜也辣 → 你也不喜欢湘菜 → 建议你婉拒。

> 画面：对比 Hermes 的 flat string 记忆

Hermes 和 OpenCLAW 的记忆是 flat string——
记住了「你不吃川菜」，
但不会自动关联到湘菜。
只有 Xuanji 的拓扑图，能让 LLM 做真正的**多跳推理**。

---

### 场景 4：异步——你不必等（2:35 - 3:20）

> 画面：分屏——左边是 Claude Code 等待转圈，右边是 Xuanji 继续聊天

**旁白**：
还有一个很多人忽略的事：**你不必等**。

用 Claude Code 或 Cursor 做重构，
你会看到 spinning cursor 转啊转，
你什么都做不了。

Xuanji 把任务派出去之后——你继续聊天。
它自己会安排 worker 在后台干活。

> 画面：后台任务完成 → 系统 prompt 自然注入 → Xuanji 说「之前派的任务完成了」

而且更好玩的是，它完成之后**会主动回来告诉你**。
用一种很自然的语气：
「哦对了，刚才那个重构我弄完了，结果是……」
不是弹个通知，是 LLM 自己「想起」这件事。

Hermes 的 background 命令能做到后台运行，
但做完就完了——不会主动汇报。
OpenCLAW 也一样。
这是 Xuanji 独有的**完成通知链路**。

---

### 场景 5：自学习——学会了就能直接用（3:20 - 3:55）

> 画面：learn SpringBoot → 提取 entities → 自动安装 MCP → 生成 Skill

**旁白**：
Xuanji 还有一个独特的能力——自学习。

你告诉它：「学一下 SpringBoot 3」。
它会自己搜索文档、提取关键实体和概念、
按需安装 MCP 工具、
最后生成一个可以直接用的 Skill。

没有「先装插件、再配置、再重启」的步骤。
一步到位。

**对比一下**：
Hermes 也有 skill 系统，但依赖在线 Hub，
安装需要 `hermes skills install` 命令。
OpenCLAW 的 .agents 目录也需要手动配置。
Xuanji 的全自动学习链路——是唯一做到「说学就学，学完就用」的。

---

### 场景 5+：扩展性——你的 Xuanji，你的规则（3:55 - 4:20）

> 画面：三路并行动画——左边编辑 YAML prompt，中间 `install` MCP 服务器，右边加载 Skill

**旁白**：
Xuanji 还有一个常常被低估的优势——**扩展深度**。

它不是「出厂配好，用户只能接受」的封闭系统。

你想优化它的行为？直接改 YAML 文件。
L0、L1、L2 各层的 prompt 都是独立 YAML，
你想给「写代码」场景加一条规则——
打开 `l1-write-code.yaml`，加一行就行。
不用改代码，不用重新编译。

你想给它新的能力？接入 MCP 服务器。
`install("PostgreSQL")` → 搜索 → 安装 → hot-load。
**不需要重启**，立即可用。

你想保存一个工作流？生成 Skill。
学完了 SpringBoot，生成的 Skill 可以直接复用。
下次说「帮我搭个 SpringBoot 项目」——它已经会了。

对比一下：
- Hermes 改 prompt 要改 Python 代码，加 MCP 要 reset session
- OpenCLAW 的 .agents 只覆盖 agent 级别的 prompt，没有分层结构

**Xuanji 的玩法是：你买的是毛坯房，不是精装房。**
你想要的每一个能力——改 prompt、接 MCP、写 Skill——都不需要碰源代码。

---

### 场景 6：那 Xuanji 的代价是什么？（4:20 - 4:55）

> 画面：三栏图表——Xuanji、Hermes、OpenCLAW，标注复杂度、社区规模

**旁白**：
当然，这些能力不是白来的。

Xuanji 的代价是：
- 架构最复杂。L0/L1/L2/L3 四层 prompt 系统，ACP 子进程池，8 表记忆系统——这是要在代码里和 31 个 YAML 文件之间跳舞。
- 没有在线 Skill Hub。你没法像 Hermes 那样 `hermes skills browse`。
- 没有 15 个消息平台的网关。只有桌面和 CLI。
- 社区还很小。

Hermes / OpenCLAW 走的是另一条路：
简单的单层 prompt，简单的 KV 记忆，简单的 delegate。
但换来的是什么？最大的开源 AI agent 社区：373k stars。
20+ provider 支持，15+ 平台网关。

**所以这个选择很明确——**
Xuanji 选了「每个模块做到极致」，
Hermes 选了「最广的覆盖和最成熟的生态」。
如果你想要最深的 agent 协作和记忆——选 Xuanji。
如果你要大生态和多平台——Hermes 更好。
