# Xuanji 项目后续优化建议

## 📋 优化方向

### 1. 测试覆盖率提升 ⭐⭐⭐⭐⭐

**当前状态**: 缺少系统化的测试

**建议**:
- 为重构后的核心模块编写单元测试
- 重点测试：
  - DependencyContainer（依赖注入）
  - MiddlewarePipeline（中间件执行）
  - EventBus/MessageBus（事件/消息传递）
  - ConfigService（配置合并）
  - SQLiteStorage（存储操作）

**优先级**: 🔥 高
**预计工作量**: 2-3 天
**收益**: 保证重构代码的正确性，防止回归

---

### 2. 性能优化 ⭐⭐⭐⭐

#### 2.1 消息历史管理优化
**问题**: MessageManager 保存所有历史消息，可能导致内存占用过大

**建议**:
- 实现消息分页加载
- 添加消息淘汰策略（保留最近 N 条）
- 使用流式存储，避免全量加载

#### 2.2 权限决策缓存优化
**问题**: DecisionStore 每次查询都访问数据库

**建议**:
- 添加内存缓存层（LRU Cache）
- 批量预加载常用决策
- 使用 Redis 作为分布式缓存（如果需要）

#### 2.3 记忆检索性能优化
**问题**: MemoryStore 的向量搜索可能较慢

**建议**:
- 使用 HNSW 索引加速向量搜索
- 实现增量索引更新
- 添加查询结果缓存

**优先级**: 🔥 中高
**预计工作量**: 3-5 天
**收益**: 提升系统响应速度，降低资源消耗

---

#### 3.3 分布式追踪

### 4. 错误处理增强 ⭐⭐⭐⭐

#### 4.1 统一错误类型
**问题**: 错误类型分散，难以统一处理

**建议**:
```typescript
// 定义错误基类
abstract class XuanjiError extends Error {
  constructor(
    message: string,
    public code: string,
    public details?: any
  ) {
    super(message);
    this.name = this.constructor.name;
  }
}

// 具体错误类型
class ToolExecutionError extends XuanjiError {}
class PermissionDeniedError extends XuanjiError {}
class ConfigurationError extends XuanjiError {}
class StorageError extends XuanjiError {}
```

#### 4.2 错误恢复策略
**建议**:
- 实现自动重试机制（已有 RetryMiddleware，需完善）
- 添加降级策略（如 LLM 调用失败时使用缓存）
- 实现断路器模式（防止级联失败）

#### 4.3 错误上报
**建议**:
- 集成 Sentry 或类似工具
- 自动上报未捕获异常
- 添加错误聚合和告警

**优先级**: 🔥 中高
**预计工作量**: 2-3 天
**收益**: 提升系统稳定性，快速发现问题

## 💡 建议

1. **优先解决痛点**: 先解决当前最影响开发和使用的问题
2. **渐进式优化**: 不要一次性做太多，分阶段执行
3. **持续迭代**: 每个阶段完成后评估效果，调整计划
4. **保持文档更新**: 每次优化都更新相关文档
